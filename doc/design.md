## 1. 개요

GitKraken을 목표 모델로 하는 데스크톱 Git 클라이언트를 설계한다.
CLI를 몰라도 Git의 전체 워크플로를 수행할 수 있어야 하고, 커밋 히스토리를
그래프로 직관적으로 보여주며, 원격 서버와의 통신이 체감상 빨라야 한다.

### 1.1 목표 (Goals)

| # | 목표 | 성공 기준 |
|---|------|-----------|
| G1 | 일상 Git 워크플로의 GUI 완결 | clone/commit/branch/merge/rebase/stash/tag/remote를 CLI 없이 수행 |
| G2 | 대규모 저장소에서의 반응성 | 커밋 10만 개 저장소에서 그래프 초기 렌더 < 1.5s, 스크롤 60fps ✅ **실측 493ms — §4.1.1.2** |
| G3 | 원격 통신 비용 최소화 | 반복 작업(fetch/pull)의 누적 전송량을 표준 git 대비 유의미하게 절감. 구체 수치는 Phase 3 실측 후 확정 |
| G4 | UI 무정지 | UI 스레드의 단일 블로킹 구간 ≤ 50ms (§3.3 예산 — 초안의 "0ms"는 측정 불가능한 절대 언어라 교체) |
| G5 | 크로스 플랫폼 | Windows / macOS / Linux 단일 코드베이스 |

### 1.2 비목표 (Non-Goals)

- 자체 Git 서버 구현 — 클라이언트에 집중한다.
- Git 프로토콜의 재구현 — 검증된 구현(libgit2, git CLI)을 활용한다.
- 이슈 트래커 / CI / 코드 리뷰 통합 — v1 범위 밖. v2 이후 플러그인으로 검토.
- 웹/모바일 클라이언트.

### 1.3 대상 사용자

1. **입문자** — CLI가 부담스러워 GUI로 Git을 배우려는 개발자.
2. **숙련자** — 히스토리 탐색·인터랙티브 rebase·충돌 해결처럼 시각화 이득이 큰 작업에서만 GUI를 쓰는 개발자.
3. **모노레포 사용자** — 저장소가 커서 clone/fetch 비용이 실제 병목인 팀.

### 1.4 운영 환경 전제 (중요)

**이 프로젝트의 설계 결정 다수가 아래 환경 특성에서 도출된다.**

| 자원 | 상태 | 설계에 미치는 영향 |
|------|------|-------------------|
| **네트워크 대역폭 / 트래픽 총량** | **제약이 크다** | 최우선 최적화 대상. 전송 바이트 1KB가 CPU 1초보다 비싸다. |
| **컴퓨팅 파워 (CPU / 메모리 / 디스크)** | 충분하다 | CPU를 써서 전송량을 줄일 수 있다면 **항상 그렇게 한다.** |
| **오프라인 작업 빈도** | 잦다 | 네트워크가 없는 상태에서도 전체 히스토리 탐색이 가능해야 한다. |

**여기서 나오는 세 가지 원칙**

1. **압축은 항상 최대로.** 일반적인 Git 튜닝은 "빠른 회선에서는 압축을 낮춰라"고
   조언하지만, 이 환경에서는 반대다. CPU가 남아도는데 바이트가 비싸므로
   압축 수준을 최대로 고정한다. ([performance.md](performance.md) §6.4)

2. **한 번 받은 것은 다시 받지 않는다.** 지연 로딩(partial clone)은 전송을
   나중으로 미룰 뿐이고, 같은 객체를 반복 요청하면 **총 트래픽은 오히려 늘어난다.**
   오프라인 작업이 잦은 이 환경에서는 "한 번에 다 받고 이후 증분만 받는" 쪽이
   누적 트래픽 기준으로 유리하다. 이것이 §2.4의 clone 기본값 결정 근거다.

3. **로컬 계산으로 대체 가능한 통신은 전부 대체한다.** 캐시, 인덱스, commit-graph
   생성 등 CPU·디스크를 쓰는 작업은 비용이 사실상 0이므로 아낌없이 사용한다.

### 1.5 잠정 결정 표기

일부 결정은 **실측 데이터가 없는 상태의 잠정안**이다. 문서 내에서 🔶 로 표시하며,
Phase 3의 벤치마크 결과에 따라 변경될 수 있다. 변경 시 이 문서와
[performance.md](performance.md)를 함께 갱신하고, 변경 근거를 §12 결정 기록에 남긴다.

## 2. 기술 스택 결정

### 2.1 최종 선택

| 영역 | 선택 | 버전 |
|------|------|------|
| 언어 | Python | 3.12+ |
| UI 프레임워크 | PySide6 (Qt 6) | 6.7+ |
| Git 로컬 엔진 | pygit2 (libgit2 바인딩) | 1.15+ |
| Git 네트워크 엔진 | git CLI (subprocess) | 2.40+ |
| 비동기 | QThreadPool + QRunnable (+ asyncio는 I/O 전용) | - |
| 설정/앱 상태 | QSettings (Qt 내장) | - |
| 캐시 (commit_meta·remote_stats, Phase 3+) | SQLite (stdlib `sqlite3`) | - |
| 패키징 | PyInstaller + 플랫폼별 인스톨러 | - |
| 테스트 | pytest, pytest-qt | - |

### 2.2 선택 근거

**왜 Python + PySide6 인가**

- 요청된 기본 언어를 유지하면서, Qt의 성숙한 데스크톱 위젯 생태계를 얻는다.
- `QAbstractItemModel` + `QTableView`의 **뷰포트 가상화**가 기본 제공된다.
  커밋 10만 개를 모델에 올려도 실제로 그려지는 행은 화면에 보이는 40~50개뿐이다.
  이것이 G2를 달성하는 핵심 장치다.
- `QGraphicsView` / 커스텀 `QStyledItemDelegate`로 GitKraken 스타일의 커밋 그래프를
  직접 그릴 수 있다.
- LGPL 라이선스라 상용 배포에도 (동적 링크 조건 하에) 문제가 없다.

**Python의 약점과 대응**

| 약점 | 대응 |
|------|------|
| GIL로 인한 CPU 병렬성 제약 | 무거운 작업은 C 레벨에서 GIL을 해제하는 pygit2에 위임하거나, `subprocess`로 별도 프로세스에 넘긴다. ~~diff 파싱이 Python CPU 병목~~ — **정정(Phase 1 실측)**: diff 분해는 libgit2(C)가 수행하고 Python은 얕은 매핑만 한다. 실측된 Python CPU 비용은 레인 배치(행당 5.3µs, §4.1.1.2)다. Rust(`pyo3`) 교체 후보는 diff 파서가 아니라 레인 배치와 (미래의) 단어 단위 diff이며, 병목이 측정으로 확인된 뒤에만 착수한다. |
| 배포 용량 (~80MB) | Tauri(~10MB)보다 크지만 Electron(~150MB)보다 작다. 수용 가능. |
| 시작 속도 | 지연 임포트 + 스플래시 화면. 목표 콜드 스타트 < 2s. |

**탈락한 선택지**

- *Rust + Tauri*: 성능·용량은 최상이나 단일 개발자 기준 개발 속도가 3~4배 느리다.
  성능 병목이 실제로 Python에서 발생하는 것이 측정으로 확인되면 그 모듈만 Rust로 옮긴다.
- *Electron*: GitKraken의 실제 스택이지만 파이썬 요구사항에서 벗어나고 리소스가 무겁다.
- *Python + 웹 프론트*: 로직/UI 분리는 깔끔하나 파일 드래그, OS 자격증명 저장소 연동,
  네이티브 컨텍스트 메뉴 같은 데스크톱 통합에서 손해가 크다.

### 2.3 핵심 판단: pygit2 / git CLI 하이브리드

이 설계의 가장 중요한 기술적 결정이다. **로컬 작업과 네트워크 작업에 서로 다른 엔진을 쓴다.**

```
로컬 읽기/쓰기          →  pygit2 (libgit2)
네트워크 통신           →  git CLI (subprocess)
```

**pygit2를 로컬에 쓰는 이유**

- 프로세스 생성 비용이 없다. 커밋 목록을 훑을 때 `git log`를 매번 띄우면
  Windows에서 프로세스당 20~40ms가 소모된다. 인메모리 순회는 이 비용이 0이다.
- 결과가 파싱된 객체로 온다. 텍스트 출력 파싱은 로케일/개행/이름에 포함된
  특수문자 등에서 깨지기 쉽다.
- ODB(Object Database)에 직접 접근해 diff, blame, revwalk를 세밀히 제어할 수 있다.

**네트워크를 git CLI에 맡기는 이유** (통신 속도 요구사항의 근거)

- **protocol v2**: 참조(ref) 광고량을 극적으로 줄이는 v2를 git CLI는 기본 지원하지만,
  libgit2의 v2 지원은 아직 부분적이다. ref가 수만 개인 저장소에서 이 차이는 크다.
- **partial clone** (`--filter=blob:none`): libgit2는 미지원. G3 달성의 핵심 수단이므로
  CLI가 필수다.
- **credential helper**: OS 자격증명 저장소(Windows Credential Manager, macOS Keychain),
  GitHub CLI 토큰, SSH agent 연동을 git이 이미 처리해준다. 직접 구현할 이유가 없다.
- **SSH 멀티플렉싱**: `ControlMaster`를 통한 연결 재사용은 OpenSSH 기능이며
  CLI 경로에서만 자연스럽게 동작한다.

**환경 특성에 비춘 재검토 결과: 하이브리드를 유지한다.**

§1.4대로 네트워크가 병목이고 컴퓨팅은 여유롭다면, 엔진 선택 기준은
"어느 쪽이 전송 바이트를 더 줄일 수 있는가" 하나로 수렴한다.

| 판단 항목 | libgit2 단독 | git CLI 경유 |
|-----------|-------------|-------------|
| protocol v2 (ref 광고량 축소) | 부분 지원 | 완전 지원 |
| partial clone / 필터 | 미지원 | 지원 |
| 협상 알고리즘 선택 | 불가 | 가능 |
| 압축 수준 세밀 제어 | 제한적 | 가능 |
| 프로세스 생성 오버헤드 | 없음 | 있음 (수십 ms) |

CLI의 유일한 단점인 프로세스 생성 비용은 **CPU 자원의 문제**이고,
이 환경에서 CPU는 남아돈다. 반면 CLI만이 제공하는 기능들은 전부
**전송 바이트를 직접 줄이는 수단**이다. 트레이드오프가 명백히 한쪽으로 기운다.

따라서 네트워크 경로는 git CLI로 확정한다. 로컬 읽기를 pygit2로 두는 이유는
통신과 무관하게 UI 반응성(G2) 때문이며, 이 역시 유지한다.

자세한 내용은 [performance.md](performance.md) 참조.

### 2.4 Clone 기본값: 전체 clone 🔶

**결정: `--filter` 없는 전체 clone을 기본값으로 한다. partial clone은 선택지로만 제공한다.**

일반적인 "전송량 최소화" 논리만 보면 partial clone이 정답처럼 보인다.
하지만 §1.4의 환경에서는 결론이 뒤집힌다.

| 관점 | 전체 clone | partial clone |
|------|-----------|---------------|
| **초기** 전송량 | 크다 | 작다 |
| **누적** 전송량 | 초기 1회 + 증분만 | 초기 + blob 요청이 계속 누적 |
| 같은 객체 재요청 가능성 | 없음 | 있음 (GC로 지연 객체가 정리된 경우 등) |
| 오프라인에서 과거 파일 열람 | 가능 | **불가** |
| 요청당 오버헤드 | 1회 | blob 요청마다 HTTP 왕복 + 헤더 |

**핵심 논리**: 트래픽 총량이 제한된 환경에서 중요한 것은 초기 전송량이 아니라
**누적 전송량**이다. partial clone은 전송을 미룰 뿐 없애지 않으며,
히스토리 탐색이 잦으면 잘게 쪼개진 요청이 오버헤드와 함께 누적된다.
게다가 오프라인 작업이 잦다는 조건은 지연 로딩과 정면으로 충돌한다.

**여전히 partial clone을 제공하는 경우**

- 저장소가 매우 커서 초기 clone 자체가 트래픽 예산을 초과하는 경우
- 특정 하위 디렉터리만 잠깐 확인하면 되는 일회성 작업
- 오프라인 요구가 없는 저장소

clone 다이얼로그에서 예상 전송량과 함께 선택지를 제시하고,
**기본 선택은 전체 clone**으로 둔다. 선택 시 오프라인 제약을 명시한다.

> 🔶 이 결정은 Phase 3에서 실제 사용 패턴(히스토리 탐색 빈도, 저장소 크기 분포)을
> 측정한 뒤 재검토한다. 저장소가 예상보다 크고 과거 blob 접근이 드물다면
> `blob:limit=<크기>` 같은 절충안(대용량 파일만 지연)이 더 나을 수 있다.

## 3. 아키텍처

### 3.1 레이어 구조

```
┌──────────────────────────────────────────────────────────┐
│  Presentation Layer  (PySide6 위젯 / 델리게이트 / 다이얼로그)  │
│  - 절대 블로킹 호출 금지. 시그널만 주고받는다.                    │
├──────────────────────────────────────────────────────────┤
│  ViewModel Layer  (QAbstractItemModel 파생 + 상태 스토어)     │
│  - UI가 소비할 형태로 도메인 데이터를 가공/캐싱                   │
├──────────────────────────────────────────────────────────┤
│  Application Layer  (유스케이스 / 작업 큐 / 트랜잭션)            │
│  - "브랜치 병합", "원격에 푸시" 같은 단위 작업을 조율             │
├──────────────────────────────────────────────────────────┤
│  Domain Layer  (Repository, Commit, Ref, Diff … 순수 모델)   │
│  - Qt/pygit2 의존성 0. 순수 파이썬. 테스트가 가장 쉬운 층.        │
│  - Infrastructure와 Presentation이 이 층에 의존한다 (역방향 아님) │
├──────────────────────────────────────────────────────────┤
│  Infrastructure Layer  (Domain을 향해 의존 ↑)                │
│  ├─ LocalGitEngine   (pygit2)                            │
│  ├─ RemoteGitEngine  (git CLI subprocess, Phase 3+)       │
│  ├─ CacheStore       (SQLite, Phase 3+ — Phase 1 설정은     │
│  │                    QSettings로 충분, ADR-12)             │
│  └─ CredentialBridge (git credential helper 위임, Phase 3+) │
└──────────────────────────────────────────────────────────┘
```

**의존은 항상 Domain을 향한다 (Dependency Rule).** 상위 층(Presentation·
ViewModel·Application)도, 다이어그램 최하단의 Infrastructure도 모두 Domain에
의존하며, Domain은 어떤 층도 import하지 않는다. Infrastructure는 I/O 기반이라
시각적으로 맨 아래에 두었지만, pygit2 타입을 Domain 객체로 변환해 내보내므로
실제 의존은 Domain을 향해 '위로' 흐른다(`local_engine.py`가
`domain.models`/`domain.errors`를 import). 이 규칙 덕분에 나중에 Infrastructure를
Rust 확장으로 교체하거나 UI를 다른 프레임워크로 바꿔도 Domain/Application은
그대로 남는다.

> 초안의 "의존 방향은 항상 아래로만 흐른다"는 문구는 이 다이어그램 배치와
> 모순되어 정정했다. 코드는 처음부터 Dependency Rule을 따르고 있었다.

### 3.3 동시성 모델

**입력 크기에 비례해 커질 수 있는 Git 작업은 UI 스레드에서 실행해선 안 된다.**
커밋 순회는 커밋 수에, ref 열거는 ref 수에, diff는 변경 크기에 비례한다 —
전부 워커 대상이다. 반면 상한이 알려진 짧은 호출(저장소 핸들 열기 ~20ms 등)은
G4 예산 안에서 UI 스레드 직접 호출을 허용한다. 매 읽기를 유스케이스로 감싸는
것은 빈 의례다.

**G4 예산**: UI 스레드의 단일 블로킹 구간은 **50ms를 넘지 않는다.**
(초안의 "블로킹 0ms"는 측정 불가능한 절대 언어라 예산으로 교체 — 사람이
지연을 인지하는 문턱과 실측된 고정비를 근거로 한 값이다.)

```
[UI 스레드]                    [워커 풀 (QThreadPool)]
    │                                  │
    │  usecase.execute(params) ───────►│
    │                                  │  pygit2 호출 / subprocess 실행
    │  ◄──── progress(percent, msg) ───│  (진행 중 주기적 시그널)
    │  ◄──── finished(result) ─────────│
    │  ◄──── failed(error) ────────────│
```

위 다이어그램은 **장시간·쓰기·원격 작업의 계약**이다. 예산 내 짧은 읽기까지
이 경로를 강제하는 것은 아니다.

**규칙**

1. 모든 유스케이스는 `QRunnable`을 상속하고 `signals` 객체로만 결과를 전달한다.
2. 워커는 Qt 위젯을 직접 만지지 않는다. 시그널/슬롯의 큐 연결이 스레드 경계를 넘겨준다.
3. **쓰기 작업은 저장소 단위로 직렬화한다.** 같은 저장소에 merge와 checkout이
   동시에 들어가면 인덱스가 깨진다. 저장소별 작업 큐가 쓰기 작업을 하나씩 처리한다.
   (Phase 2에서 `application/write_queue.py`로 구현 — FIFO, 실패해도 큐는
   멈추지 않음, 작업마다 전용 엔진 핸들)
4. **읽기 작업은 병렬 허용.** diff 계산, blame, 히스토리 조회는 동시에 여러 개 돌아도 된다.
   단, 워커마다 **자신만의 저장소 핸들**을 연다 — libgit2 핸들은 스레드 간 공유가
   안전하지 않다.
5. 모든 장시간 작업은 취소 가능해야 한다. 취소 지점은 작업 종류에 따라 다르다:
   subprocess는 프로세스 트리 종료, fetch/clone/push/checkout은 pygit2 진행 콜백의
   중단 예외. **revwalk는 진행 콜백이 없으므로** 커밋 경계에서 플래그를 확인하는
   방식이며, 정렬 순회의 선행 구간(§4.1.1.1의 83%)은 중단 지점이 없다 —
   팩된 저장소 실측 최악값(~600ms)은 종료 대기 한도(2s) 안이다.

## 4. 주요 컴포넌트 설계

### 4.1 커밋 그래프

GitKraken의 정체성이자 이 프로젝트에서 가장 까다로운 부분이다.

**데이터 흐름**

```
pygit2.Repository.walk()  →  Commit 스트림
        ↓
레인 배치 알고리즘 (domain/graph.py)  →  각 커밋에 (lane, color, edges, lane_count, is_merge) 부여
        ↓
CommitGraphModel (QAbstractTableModel)
        ↓
QTableView + GraphDelegate  →  QPainter로 선/노드 직접 그리기
```

**레인 배치 알고리즘 개요**

시간 역순으로 커밋을 순회하며 "활성 브랜치 라인" 목록을 유지한다.

1. 현재 커밋이 어떤 활성 레인의 다음 대상이면 그 레인을 차지한다.
   아니면 비어 있는 가장 왼쪽 레인을 새로 할당한다.
2. 부모가 1개면 그 부모를 같은 레인의 다음 대상으로 예약한다 (직선).
3. 부모가 2개 이상이면(머지 커밋) 첫 부모는 현재 레인을 잇고,
   나머지 부모는 새 레인으로 분기시킨다 (곡선 엣지).
4. 어떤 레인의 대상 커밋이 이미 다른 레인에서 처리되었으면 그 레인은 병합되어 닫힌다.
5. 레인 색상은 **라인 할당 순서 기반**이다: 새 라인이 열릴 때마다 팔레트의
   다음 색을 부여하고, 그 라인이 살아있는 동안 유지한다. 초안의 "레인 인덱스
   기반"은 구현과 다르게 서술된 것을 정정했다 — 인덱스 기반이면 레인이
   재사용될 때(예: 한 히스토리가 끝나고 무관한 히스토리가 같은 레인을 이어받을 때)
   서로 다른 라인이 같은 색이 되어 하나로 오독된다. 할당 순서 기반은 이를 막으며,
   `TestOrphanBranch`가 이 동작을 고정한다.

**경계 사례 규칙** (5단계가 다루지 않는 상황 — 구현이 확립했고 테스트로 고정됨)

- **이미 배치된 부모**: 부모가 자식보다 먼저 나온 경우(시간순 정렬의 시계 오차,
  rebase 흔적) 그 부모를 기다리는 레인을 열지 않고 현재 레인을 닫는다.
  그러지 않으면 영영 닫히지 않는 레인이 남아 이후 모든 행이 밀린다.
- **범위 밖 부모**: shallow 경계 너머의 부모도 나가는 엣지는 그린다 —
  히스토리가 끊긴 게 아님을 시각적으로 보여주기 위해서다.
- **공유 부모 합류**: 다른 레인이 이미 같은 부모를 기다리고 있으면 새 레인을
  열지 않고 그 레인에 합류시킨다 (레인 중복 방지).
- **머지 레인 재사용**: 머지로 방금 닫힌 레인을 새 분기가 우선 재사용해
  그래프가 옆으로 덜 퍼지게 한다 (1단계 "가장 왼쪽 빈 레인"의 정제).

**렌더 계약**: 각 행의 `lane_count`(= 행 위/아래 경계에 필요한 레인 수의 최댓값)와
`is_merge`는 델리게이트의 폭 계산과 노드 모양 결정에 쓰인다. 이 두 값은
알고리즘의 부산물이 아니라 UI 층과의 계약이다.

**성능 설계** (Phase 1 실측으로 개정됨 — §4.1.1 참조)

- **백그라운드 로딩**: 커밋 순회는 워커 스레드에서 수행하고, 500개 묶음 단위로
  UI에 전달한다. UI 스레드는 배치 계산과 렌더링만 담당한다.
- **결과 캐싱**: ~~계산된 레인 배치를 SQLite에 저장해 재사용한다.~~
  **철회.** G2를 이미 충족하므로 근거 없는 복잡도가 된다. (§4.1.1.2)
- **commit-graph 활용**: 순회 가속 효과는 실측 오차 수준으로 판명(§4.1.1.3).
  자동 생성은 하지 않는다. `--changed-paths`의 경로 조회 가속만 잠정 후보로
  남기며, 경로 히스토리 기능(`git log -- path` 상당)을 만들 때 벤치로 확인 후
  도입한다. 🔶

#### 4.1.1 Phase 1 측정 결과

> **이 절은 한 차례 잘못된 결론을 담았다가 정정되었다.**
> 최초 측정에서 "커밋 순회는 커밋당 약 0.23ms의 고유 비용을 가지며 G2는 미달"이라고
> 결론지었으나, 이는 **벤치마크 저장소가 비현실적으로 구성된 탓**이었다.
> 아래 §4.1.1.2가 정정된 내용이다. 잘못된 판단이 어떻게 나왔는지는
> §4.1.1.3에 남긴다 — 같은 실수를 반복하지 않기 위해서다.

##### 4.1.1.1 확인된 사실: 정렬된 순회는 앞에서 비용을 몰아 치른다

pygit2의 walker는 정렬 모드를 지정하면 첫 커밋을 내놓기 전에 순회 대상을
대부분 읽는다. 커밋 10만 개(팩된 저장소) 기준:

| 구간 | 소요 |
|------|------|
| 첫 커밋까지 | 600ms |
| 나머지 99,999개 | 124ms |

따라서 **뷰포트 단위 지연 로딩(pull)으로는 초기 표시를 앞당길 수 없다.**
비용의 83%가 첫 결과 이전에 발생하기 때문이다. 이 사실은 정정 후에도 유효하며,
밀어넣기(push) + 워커 스레드 구조를 채택한 근거로 남는다.

**적용한 설계**

- 모델은 당겨오기(pull, `canFetchMore`/`fetchMore`)가 아니라
  **밀어넣기(push, `append_commits`)** 방식이다.
- 순회는 `application.commit_loader.CommitLoader`(QRunnable)가 워커 스레드에서
  수행하고, 500개 묶음마다 시그널로 UI에 넘긴다.
- 워커는 **자신만의 저장소 핸들을 연다.** libgit2 핸들을 UI 스레드와 공유하면
  사용자가 diff를 여는 동안 같은 핸들을 동시에 쓰게 된다.

##### 4.1.1.2 G2 현황: 충족

커밋 10만 개 저장소(팩됨 + commit-graph, `git fast-import`로 생성)에서 측정:

| 항목 | 측정값 | 목표 | 평가 |
|------|--------|------|------|
| `open_repository()`의 UI 스레드 점유 | 20.9ms | — | G4 충족 |
| **첫 행이 화면에 나타나기까지** | **493ms** | 1,500ms | **G2 충족** |
| 전체 10만 행 로딩 완료 | 1,822ms | — | — |
| 그중 UI 스레드가 레인 배치에 쓴 시간 | 532ms (29%) | — | 아래 참조 |

**레이아웃 캐시는 G2를 위해 필요하지 않다.** 목표를 이미 3배 여유로 만족하므로,
SQLite 캐시는 근거 없는 복잡도가 된다. 착수하지 않는다.

**남은 개선 여지** (G2와 무관, 우선순위 낮음)

- 로딩 중 UI 스레드가 29%를 레인 배치에 쓴다(행당 5.3µs). 화면이 멈추지는 않지만,
  10만 행 로딩 중 타이머·애니메이션이 지연될 수 있다. 레인 배치를 워커 스레드로
  옮기면 해소되나, 현재 체감 문제가 보고된 바 없어 착수하지 않는다.
- 느슨한(loose) 오브젝트가 많은 저장소는 순회가 40배 느리다(아래 참조).
  git이 `gc`로 자동 관리하는 영역이므로 클라이언트가 개입하지 않는다.
  다만 저장소를 열 때 느슨한 오브젝트 수가 비정상적으로 많으면
  `git gc` 실행을 제안하는 것은 검토할 만하다.

##### 4.1.1.3 정정 경위 — 벤치마크가 만든 가짜 결론

최초 측정은 pygit2의 `create_commit`을 2만 번 호출해 만든 저장소를 대상으로 했다.
이 방식은 커밋을 **느슨한 오브젝트 파일 2만 개**로 남긴다. 실제 저장소는
clone하면 팩된 상태로 오고 이후 `git gc`가 자동으로 팩을 유지하므로,
이 fixture는 현실을 대표하지 않았다.

내용이 동일한 저장소를 저장 형식만 바꿔 측정한 결과:

| 저장소 | 커밋당 순회 비용 | 첫 커밋까지 |
|--------|-----------------|------------|
| 20k · 느슨한 오브젝트 | 197.8µs | 3,928ms |
| 20k · 팩됨 | **4.9µs** | **77ms** |
| 20k · 팩됨 + commit-graph | 4.7µs | 76ms |
| 100k · 팩됨 + commit-graph | 7.2µs | 600ms |

**팩 압축만으로 40배 차이가 난다.** "커밋당 0.23ms"는 파싱 비용이 아니라
NTFS에서 작은 파일 2만 개를 여는 I/O 비용이었다.

commit-graph의 기여도 재해석해야 한다. 최초 측정에서 9%에 그친 것은
효과가 작아서가 아니라, 그 저장소의 병목이 파일 I/O였기 때문에
가려진 것이다. 팩된 저장소에서는 순회 자체가 이미 빨라
commit-graph의 추가 이득이 측정 오차 수준(4.9 → 4.7µs)으로 나타난다.

**교훈**

1. 합성 벤치마크는 대상 시스템의 **저장 형식까지** 현실과 맞춰야 한다.
   커밋 개수만 맞추는 것으로는 부족했다.
2. "고유 비용"이라고 단정하기 전에 변수를 분리해야 한다. 정렬 모드 세 가지를
   비교하면서 총비용이 같길래 고유 비용이라고 결론지었으나, 세 측정 모두
   같은 병목(파일 I/O)을 공유하고 있었으므로 그 비교로는 아무것도 분리되지 않았다.
3. 목표 규모(10만)로 직접 측정하지 않고 2만에서 외삽한 것도 오류를 키웠다.
   외삽값은 18초였고 실측값은 0.5초였다.

### 4.2 Diff 뷰

- **표시 모드**: 통합(unified) / 나란히(side-by-side) 전환. (side-by-side는 Phase 2+)
- **구문 강조**: Pygments를 워커 스레드에서 실행하고 결과를 캐싱.
  파일이 큰 경우(> 2MB) 강조를 생략하고 플레인 텍스트로 표시. (Phase 2+)
- **가상 스크롤**: diff도 커밋 목록과 동일하게 **렌더링만 가상화**한다.
  `DiffLine` 목록 자체는 커밋 목록의 `Commit`처럼 전량 메모리에 상주한다 —
  10만 라인이라도 수십 MB 수준이라 병목이 아니며, 그리는 것은 뷰포트 행뿐이다.
  (초안의 "메모리에 전부 올리지 않는다"는 부정확한 서술을 정정. 병적으로 큰
  diff는 스트리밍이 아니라 절단(truncate)으로 다루되, 실측으로 문제가 확인될
  때만 도입한다.)
- **단어 단위 diff**: 라인 내 변경 지점을 `difflib.SequenceMatcher`로 계산해 강조.
  (Phase 2+ — §10 남은 과제에 등재)
- **부분 스테이징**: 헝크(hunk) 단위 및 라인 단위 스테이징 지원. ✅ 구현 완료
  선택된 라인으로 패치를 합성해 인덱스에 적용한다. 상세는 §4.2.1.

#### 4.2.1 부분 스테이징의 패치 합성

선택된 줄만 담은 패치 텍스트를 만들어 **인덱스에만** 적용한다
(`pygit2.Diff.parse_diff` → `repo.apply(location=INDEX)`). 워킹 트리는
건드리지 않는다. 초안이 상정한 `git apply --cached` 대신 libgit2 API를 쓰는데,
Phase 1~2의 로컬 경로는 pygit2로 처리한다는 ADR-2와 일관되고 프로세스 생성도 없다.

합성기는 `domain/patch.py`에 있다 — 순수 파이썬이라 어려운 부분을 전부
단위 테스트로 고정할 수 있다는 것이 이 배치의 이유다.

**선택 의미론은 방향에 따라 거울상이다.** 만들어진 패치는 어느 쪽이든
인덱스에 적용되므로, 컨텍스트가 인덱스 내용과 일치해야 한다.

| 원본 줄 | 스테이징(forward) | 언스테이징(reverse) |
|---------|-------------------|---------------------|
| 선택된 `+` | 유지 (`+`) | `-`로 뒤집음 |
| 미선택 `+` | **버림** | **컨텍스트로** |
| 선택된 `-` | 유지 (`-`) | `+`로 뒤집음 |
| 미선택 `-` | **컨텍스트로** | **버림** |

기준 줄 번호도 다르다 — 패치가 적용될 파일(인덱스)의 번호여야 하므로
forward는 원본 diff의 `old_start`, reverse는 `new_start`를 쓴다.
이 두 가지를 뒤바꾸면 인덱스에 없는 줄을 컨텍스트로 요구하게 되어
`hunk did not apply`로 거부된다. (구현 중 실제로 겪은 결함이다)

**안전 속성**: 합성이 틀리면 libgit2가 거부한다 — 헝크 줄 수 불일치는
`invalid patch hunk`, 컨텍스트 불일치는 `hunk did not apply`로 실패하며
인덱스는 그대로 남는다(실측 확인). 조용한 오염 경로가 없다는 것이
이 기능을 안전하게 만드는 근거다.

**의도된 의미론**: 패치를 읽은 뒤 워킹 트리가 바뀌어도 적용은 막히지 않는다.
사용자가 화면에서 고른 내용이 그대로 올라간다(`git add -p`와 동일).
반면 **인덱스**가 그 사이 바뀌면 컨텍스트 불일치로 거부된다.

**표시와 적용의 좌표 공유**: 화면의 줄과 패치의 줄은 `(헝크 인덱스, 줄 인덱스)`
좌표를 공유한다(`iter_display_rows`). 표시용과 적용용이 각자 좌표를 계산하면
사용자가 고른 것과 실제 적용되는 것이 어긋날 수 있다.

**제약 — 적대적 리뷰에서 확정된 것들**

부분 스테이징은 조용한 데이터 손상 경로가 많은 기능이다. 아래는 실측으로
확인해 **거부하도록 만든** 경우다. 반쯤 적용하는 것보다 거부가 낫다.

| 상황 | 왜 거부하는가 |
|------|--------------|
| 바이너리 파일 | 줄 개념이 없다 |
| 이름이 바뀐 파일 | 옛 경로 삭제와 새 경로 추가가 한 쌍인데 부분 패치는 한 경로만 다뤄 절반만 적용된다 |
| 여러 줄이 한 묶음으로 바뀐 곳의 일부 선택 | diff가 인접 변경을 "삭제 묶음 + 추가 묶음"으로 그룹화하므로, 일부만 고르면 컨텍스트가 된 줄이 추가줄보다 앞에 놓여 **내용 순서가 뒤바뀐다** |

마지막 항목이 특히 위험했다 — 패치 자체는 유효해서 libgit2가 받아들이므로
오류 없이 파일 내용이 재배열된다. 삭제가 한 줄뿐인 묶음은 대응이 유일해
안전하므로 계속 허용한다.

**libgit2가 잡아주지 않는 것** (헤더가 본문보다 우선하는 경우)

- 삭제된 파일을 부분 스테이징할 때 `deleted file mode` 헤더를 그대로 내면
  인덱스 엔트리가 통째로 사라진다 — 선택하지 않은 줄까지, 오류 없이.
  결과 쪽에 줄이 남으면 삭제가 아니라 수정이므로 헤더를 강등한다.
- `apply(INDEX)`는 실행 비트를 보존하지 않아 100755가 100644로 내려앉는다.
  `old mode`/`new mode` 헤더는 거부되므로, 적용 후 인덱스에서 모드를 복원한다.

**UI 계약**: 적용 후에는 diff를 반드시 다시 읽는다. 화면의 (헝크, 줄) 좌표는
적용 즉시 낡으며, 엔진은 적용 시점에 패치를 새로 읽으므로 다음 적용이
**다른 줄에 꽂힌다.** 연타 가드도 같은 이유로 필요하다.

### 4.3 작업 디렉터리 상태 감시

- `watchdog` 라이브러리로 워킹 트리를 감시한다.
- **디바운싱 필수**: 빌드나 브랜치 전환 중에는 수천 개 이벤트가 쏟아진다.
  200ms 윈도로 이벤트를 모아 한 번만 상태를 갱신한다.
- `.git/` 내부는 별도로 감시한다 (`HEAD`, `refs/`, `index` 변경 → 외부 CLI 조작 반영).
- `.gitignore` 대상 경로는 감시에서 제외해 `node_modules` 같은 디렉터리로 인한
  이벤트 폭주를 막는다.

### 4.4 인증

직접 자격증명을 저장하지 않는다. **전부 git의 credential helper에 위임한다.**

```
푸시 시도
   → git CLI 실행
   → git이 credential helper 호출 (OS 키체인 / GitHub CLI / SSH agent)
   → 자격증명 없으면 git이 프롬프트 요청
   → 클라이언트가 이를 감지해 GUI 다이얼로그 표시
   → 입력값을 helper 프로토콜로 전달, git이 저장까지 처리
```

이 방식의 이점: 보안 책임을 검증된 구현에 넘기고, 사용자가 이미 CLI에서
설정해둔 인증을 그대로 재사용한다. SSH 키 비밀번호도 agent가 처리한다.

### 4.6 원격 작업 계측 (Phase 3)

**왜 기능보다 먼저인가.** 이 프로젝트의 목적함수는 누적 전송 바이트다
(performance.md §1.1). 측정할 수 없으면 최적화의 효과를 주장할 수 없다.
§4.1.1.3에서 측정 없이 내린 결론이 이미 한 번 틀렸다.

**측정원 두 가지**

| 출처 | 얻는 것 |
|------|---------|
| stderr progress | 전송 바이트, 객체 수, 처리량, ref 갱신 |
| Trace2 이벤트(JSON) | 구간별 소요시간, 협상 왕복 횟수, 협상된 프로토콜 버전 |

파서는 `domain/instrumentation.py`에 둔다 — git을 실행하지 않고 출력만
해석하므로, 까다로운 부분(단위 환산, 캐리지 리턴으로 덮어쓰는 진행률)을
git 없이 테스트할 수 있다.

**측정을 가능하게 만든 설정: `transfer.unpackLimit=1`** 🔶

git 기본값(100)에서는 작은 fetch가 받은 팩을 통째로 풀어버려
`Receiving objects: ..., N KiB` 줄이 **아예 나오지 않는다.** 즉 일상적인
증분 fetch의 전송량이 측정 불가다. 이 값을 1로 두면 팩을 보관하므로
항상 측정된다.

부수 효과가 마침 이득이다 — 저장소가 팩된 상태를 유지하는데, §4.1.1.3에서
측정한 "팩된 저장소가 40배 빠름"과 같은 방향이다. 대가는 작은 팩이 쌓이는
것이며 유휴 시 repack이 필요하다.

> 🔶 팩 누적이 실사용에서 문제가 되는 시점은 아직 측정하지 않았다. repack
> 스케줄링을 붙이기 전에 팩 개수와 순회 성능의 관계를 확인한다.

**"측정하지 못함"과 "0바이트"는 다르다.** `received_bytes=None`은 측정
실패이지 0이 아니다. 둘을 섞으면 누적 전송량이 조용히 과소 집계되므로,
집계는 측정된 작업 수(`measured_operations`)를 함께 보고한다.

**설정은 명령 단위로만 준다.** `git -c key=value`로 넘기고 사용자의
전역/저장소 설정은 건드리지 않는다 — 앱이 환경을 오염시키면 사용자가
CLI로 돌아갔을 때 예상 밖의 동작을 만난다.

**계측 실패는 본 작업을 실패시키지 않는다.** Trace2 파싱이나 SQLite 기록이
실패해도 fetch는 성공한 것이다. 조용히 넘어가되 로그를 남긴다.

**fetch는 WriteQueue를 쓰지 않는다.** §3.3의 쓰기 직렬화는 인덱스와
워킹트리를 보호하기 위한 것인데, fetch가 건드리는 것은 원격 추적 ref뿐이다.
git 자체가 ref 갱신을 원자적으로 처리하므로 큐에 넣으면 로컬 작업만
불필요하게 막힌다.

#### 4.6.1 "전송량 0"과 "변경 없음"은 다르다

계측을 붙이고 나서 드러난 구분이다. 객체를 하나도 받지 않으면서 저장소
상태가 바뀌는 fetch가 실제로 있다.

| 상황 | 받은 객체 | 참조 변화 |
|------|----------|----------|
| 이미 가진 커밋을 가리키는 새 브랜치 | 0 | 늘어남 |
| `--prune` 삭제 | 0 | 줄어듦 |
| 진짜 변경 없음 | 0 | 없음 |

"받은 객체 수"로 화면 갱신 여부를 판단하면 앞의 두 경우가 "이미 최신"으로
표시되고 재로딩도 건너뛴다 — `.git`에는 있는 브랜치가 화면에 끝내 나타나지
않거나, 이미 지워진 원격 브랜치가 계속 남는다. 그래서 판정을 둘로 나눴다:

- `transferred_anything` — 팩을 받았는가 (전송 비용이 있었는가)
- `changed_anything` — 저장소가 바뀌었는가 (화면을 다시 그려야 하는가)

같은 이유로 **변경 없는 fetch는 `received_bytes=0`으로 기록한다.** 이건
측정 실패가 아니라 측정된 사실이다. 일상 사용에서 이쪽이 대다수라, 미측정으로
세면 "일부 작업을 측정하지 못했다"는 신호가 상시 오탐이 되어 정작 진짜 측정
공백을 가린다. 구분 기준은 `remote: Total ...` 줄의 유무다 — 원격이 팩을
광고했는데 바이트가 없으면 그건 진짜 측정 실패이므로 `None`을 유지한다.

#### 4.6.2 계측을 지키는 실행 규약

git CLI를 얇게 감싸는 것만으로는 계측이 성립하지 않았다. 적대적 리뷰에서
확인된 것들이며, 전부 명령 단위로 적용해 사용자 환경은 건드리지 않는다.

| 규약 | 없으면 생기는 일 |
|------|-----------------|
| `--` 로 원격 이름·refspec 분리 | `-`로 시작하는 원격 이름이 `--upload-pack=`으로 해석돼 **임의 명령이 실행된다** |
| `--no-recurse-submodules` | 서브모듈 전송량이 통째로 누락되고 상위 저장소 참조 수가 부풀려진다 |
| GIT_DIR 계열 환경변수 제거 | 상속된 `GIT_DIR`이 `-C`를 이겨 **다른 저장소로 fetch가 나가고** 계측은 이쪽에 기록된다 |
| `credential.interactive=false` + `GIT_ASKPASS=""` + `stdin=DEVNULL` | 자격증명 GUI가 워커를 무기한 붙잡는다 (실측: 12초+ → 0.6초 실패) |
| 타임아웃 시 프로세스 **트리** 종료 | 자손이 파이프를 물고 있어 타임아웃이 상한 노릇을 못 한다 (실측: 3초 지정에 40초 대기 + 고아 프로세스) |

마지막 항목이 특히 반직관적이었다. `subprocess.run(timeout=)`은 직계 자식만
죽이는데, Windows에서는 kill 직후 **타임아웃 없는** `communicate()`를 다시
호출한다. git이 띄운 `git-remote-https`나 `ssh`가 stderr 파이프를 상속한 채
살아 있으면 그 read가 반환하지 않는다. 즉 타임아웃 값이 상한이 아니었다.

세 번째 항목은 계측 관점에서 가장 나쁜 유형이다 — 실패도 경고도 없이 **다른
저장소의 바이트가 이 저장소 앞으로 기록된다.** 과소 집계는 집계표에서 눈에
띄지만 오귀속은 그렇지 않다.

### 4.7 push와 pull (Phase 3 증분 2)

#### 4.7.1 방향이 생기면서 드러난 것

fetch만 있을 때는 "전송량 = 받은 바이트"였다. push가 들어오면서 모델이
갈라졌다.

**`Writing objects`가 push의 측정원이다.** 형태는 `Receiving objects`와 같지만
방향이 반대다. 목적함수는 방향을 가리지 않으므로 누적 집계는 둘을 합산한다
(`billed_bytes`). 다만 **해당 없는 방향은 0이 아니라 `None`으로 둔다** —
push 행의 `received_bytes`를 0으로 채우면 "측정된 0바이트"와 구분되지 않는다.

**참조 줄의 화살표는 의미가 뒤집힌다.**

```
fetch  " * [new branch]  side -> origin/side"    원격 브랜치 → 로컬 추적 ref
push   " * [new branch]  main -> main"           로컬 브랜치 → 원격 브랜치
push   " - [deleted]     feature"                화살표가 아예 없다
```

그래서 `RefUpdate`의 필드를 `local_ref`/`remote_ref`가 아니라 방향 중립인
`source`/`dest`로 둔다 — 전자로 이름 붙이면 한쪽에서 반드시 거짓말이 된다.
삭제 줄에 화살표가 없는 것도 실측으로만 알 수 있었던 부분이고, 화살표를
필수로 두면 push 삭제가 "0개 참조 갱신"으로 보고된다.

**"팩 없음"의 신호도 작업마다 다르다.** fetch는 팩이 없으면 `Total` 줄 자체가
없고(`None`), push는 `Total 0 (delta 0)`을 내놓는다(`0`). 둘 다 0바이트로
기록해야 하므로 판별자가 두 가지다.

#### 4.7.2 pull — 네트워크와 로컬 쓰기가 만나는 지점

`git pull`에 통째로 맡기지 않는다. ADR-2의 경계를 지켜 **전송은 CLI, 병합은
pygit2**로 쪼갠다. 그래야 전송 절반을 fetch와 똑같이 계측하고, 쓰기 절반만
WriteQueue에 태울 수 있다.

```
PullWorker (워커 스레드)      →  fetch. 계측. WriteQueue 밖 (ADR-17)
      ↓ finished
MainWindow (UI 스레드)        →  merge_preview로 무엇이 필요한지 본다
      ↓ 빨리 감기가 필요하면
WriteQueue (직렬화된 쓰기)     →  fast_forward. 워킹트리·인덱스를 바꾼다
```

워커가 직접 합치면 같은 저장소에 쓰기 스트림이 둘 생겨 §3.3 규칙 3이 깨진다.

**빨리 감기의 순서가 중요하다.** checkout이 먼저, 참조 이동이 나중이다.

| 순서 | 결과 |
|------|------|
| 참조 이동 → checkout | **워킹트리가 갱신되지 않는다. 오류 없이.** 수정도 삭제도 반영되지 않고 인덱스에 엉뚱한 변경이 스테이징된 채 남는다 |
| `reset --hard` | 동작하지만 **커밋하지 않은 작업을 조용히 덮어쓴다** |
| **checkout → 참조 이동** | 올바르다. 더러운 트리에서는 checkout이 거부되므로 **실패해도 안전하다** — 참조가 아직 움직이지 않았고 사용자 작업도 그대로다 |

첫 번째 경로가 조용히 틀린다는 점이 위험하다. 예외가 없으므로 테스트만이
잡을 수 있고, 실제로 `tests/integration/test_merge_local.py`가 그 목적으로
존재한다.

**병합이 필요하면 시작하지 않는다** (🔶 ADR-24). 충돌 해결 UI가 Phase 4이므로,
지금 병합을 시작하면 사용자가 앱 안에서 끝낼 수 없는 상태에 갇힌다. 대신
받아온 것은 그대로 두고(트래픽을 이미 썼다) 무엇을 할 수 없는지 말한다.

#### 4.7.3 추적 대상은 설정에서 읽는다 (추측 금지)

`<origin>/<브랜치명>` 규약으로 upstream을 조합하면 두 가지가 **조용히**
깨진다. 둘 다 적대적 리뷰에서 실측으로 확인됐다.

| 상황 | 규약으로 추측했을 때 |
|------|---------------------|
| fork 워크플로 (origin=내 fork, upstream=원본) | fork의 뒤처진 브랜치를 대상으로 삼고 **"이미 최신"이라 답한다** |
| 분리된 HEAD (bisect 중) | pygit2가 `shorthand`로 `"HEAD"`를 주므로 `refs/remotes/<remote>/HEAD`가 만들어진다. 이 참조는 모든 clone에 존재하고 원격 기본 브랜치를 가리키므로 **사용자의 HEAD가 조용히 끌려간다** |

그래서 `branch.<name>.remote` / `branch.<name>.merge`를 그대로 읽는다.
분리된·unborn HEAD에서는 upstream이 없다고 답하고, pull은 가져오기만 한다.

같은 이유로 push의 `--set-upstream` 판정도 "설정이 있는가"로 한다.
"ahead/behind를 계산할 수 있는가"로 판정하면, 아직 fetch하지 않아 비교만
실패한 경우까지 미설정으로 읽어 **사용자가 지정한 추적 대상을 덮어쓴다.**

#### 4.7.4 저장소 상태를 먼저 확인한다

빨리 감기는 저장소가 평온할 때만 안전하다. 리뷰에서 확인된 것들:

| 상태 | 확인하지 않으면 |
|------|----------------|
| 병합·rebase 진행 중 | MERGE_HEAD가 살아남아 **다음 커밋이 위조된 머지 커밋**이 된다. 사용자는 파일 하나짜리 커밋을 의도했는데 히스토리에는 브랜치 전체를 병합한 커밋이 박히고, `state_cleanup`이 되돌릴 단서까지 지운다 |
| unborn HEAD (첫 커밋 전) | 워킹트리를 원격 내용으로 덮은 **뒤** 참조 생성에 실패한다. 그 상태로 커밋하면 원격과 영영 갈라지는 루트 커밋이 된다 |
| 제출 후 브랜치 전환 | 네트워크 절반이 도는 동안 사용자가 브랜치를 바꾸면 **엉뚱한 브랜치를 원격 위치로 덮어쓴다** — reflog로만 복구 가능하다 |

마지막 것은 두 WriteQueue 작업 사이의 경합이라 UI 쪽 가드로는 막을 수 없다.
제출 시점의 브랜치를 작업에 실어 보내고 실행 직전에 HEAD와 대조한다.

#### 4.7.5 실패해도 쓴 트래픽은 기록한다

push는 호출 단위가 아니라 **ref 단위로 원자적**이다. 여러 ref를 올리다 하나만
거부되면 나머지는 실제로 전송되고 원격에 반영되는데, git은 exit 1을 낸다.
예외만 던지고 끝내면 행 자체가 없어 "측정 실패"로도 잡히지 않는 **완전
무기록**이 되고, 누적 전송 바이트가 조용히 줄어든다.

그래서 실패한 작업도 계측을 만들어 예외에 실어 보내고, 워커가 기록한 뒤
예외를 그대로 전파한다. **취소 가드보다 먼저 기록한다** — 사용자가
취소했더라도 이미 나간 바이트는 목적함수에서 빠지면 안 된다.

단, 실패 경로에서는 **0 채우기를 하지 않는다.** "팩이 필요 없었다"와 "보내기
전에 끊겼다"를 구분할 수 없으므로 `None`(측정 실패)이 정직한 값이다.

#### 4.7.6 벌어진 정도를 먼저 보여준다

`git rev-list --left-right --count`로 ahead/behind를 계산해 상태바에 상시
표시한다(`원격 대비 ↑2 ↓1`). 트래픽이 비싼 환경에서 "눌러서 확인"은 비용이며,
pull과 push 중 무엇이 필요한지는 누르기 전에 알 수 있어야 한다.

## 5. UI 설계

### 5.1 메인 레이아웃

```
┌────────────────────────────────────────────────────────────────┐
│ [저장소▾]  ⟲Fetch  ↓Pull  ↑Push  ⌥Branch  ⚑Stash    🔍  ⚙   │  툴바
├────────────┬───────────────────────────────────────────────────┤
│            │                                                    │
│  LOCAL     │        커밋 그래프                                  │
│   ● main   │   ┃ ● a3f9c  로그인 버그 수정      김개발  2시간 전  │
│   ○ feat/x │   ┃╲● 8b21d  API 응답 캐싱         이코딩  5시간 전  │
│            │   ┃ ┃                                              │
│  REMOTE    │   ●╱  1f04e  Merge branch 'feat/x' 김개발  1일 전   │
│   ▸ origin │                                                    │
│            ├───────────────────────────────────────────────────┤
│  STASH     │  변경 파일          │  Diff                        │
│  TAGS      │  M src/auth.py     │  - old line                  │
│            │  A src/cache.py    │  + new line                  │
└────────────┴────────────────────┴──────────────────────────────┘
│ ⬤ main ↑2 ↓0   │  마지막 fetch: 방금  │  ▓▓▓░ Fetching… 취소   │  상태바
└────────────────────────────────────────────────────────────────┘
```

상태바의 진행 막대·취소 버튼·"마지막 fetch"는 **네트워크 작업(Phase 3)의
어피던스**다. Phase 1의 로딩은 첫 묶음 도착(실측 493ms) 즉시 조작 가능한
비차단 백그라운드 채우기라 취소 UI가 필요한 대기 상태 자체가 없다 —
`CommitLoader.cancel()`은 저장소 전환·창 닫기용 내부 메커니즘이다.
↑/↓(ahead/behind)는 로컬 계산이 가능하지만 원격 추적 ref의 신선도가
fetch에 묶이므로 Phase 3 인접 시점에 넣는다.

### 5.2 상호작용 원칙

1. **드래그 앤 드롭이 1급 조작이다.** 브랜치를 다른 브랜치 위로 끌면
   merge/rebase 선택 메뉴가 뜬다. 커밋을 브랜치 위로 끌면 cherry-pick.
2. **파괴적 작업은 항상 확인 + 되돌리기 경로 제시.**
   force push, hard reset, 브랜치 삭제 시 무엇이 사라지는지 명시하고,
   가능하면 `reflog`로 복구하는 방법을 안내한다.
3. **모든 작업은 실행된 git 명령을 로그 패널에 남긴다.**
   사용자가 GUI 조작과 CLI 명령의 대응을 배우게 된다 (교육적 가치).
4. **오류는 원문 + 해석을 함께.** "영문 원문 우선" 원칙은 git CLI를 통과하는
   오류(Phase 3 네트워크 경로)에 적용된다 — CLI의 원문이 검색 가능한 1차
   자료이기 때문이다. pygit2 읽기 경로(Phase 1)는 한국어 해석을 앞세우고
   원문은 상세 정보로 보존한다.

### 5.3 테마

- 라이트/다크 모드, OS 설정 자동 추종.
- **그래프 레인 색상 — 현재 한계를 명시한다.** 레인의 1차 구분 수단은
  색이 아니라 **가로 위치**이며, 색상(hue)은 정상 색각 사용자용 보조 단서다.
  현재 8색 팔레트는 색각 이상(CVD) 완전 구분과 명도 대비를 확보하지 못했다
  (실측: 초록/청록 쌍의 대비비 1.005:1 — 그레이스케일에서 동일). 8색을
  라이트/다크 양쪽 중간 명도로 유지하면서 CVD 완전 구분까지 만족시키는 것은
  과제약이다. Phase 5에서 CVD 시뮬레이션으로 문제 쌍을 정량화한 뒤 검증된
  CVD-safe 팔레트 또는 색 외 중복 부호화(선 스타일·마커)로 보강한다.
  눈대중 조정은 금지 — §4.1.1.3 교훈.

### 5.4 첫 실행 온보딩 — 네트워크 사용 동의 🔶

**백그라운드 prefetch는 사용자에게 묻지 않고 켜지 않는다.**

prefetch는 체감 속도를 가장 크게 개선하는 기능이지만, 동시에
**사용자가 요청하지 않은 네트워크 트래픽을 소비한다.** §1.4처럼 트래픽이
제한된 환경에서는 이것이 실질적인 비용이며, 사용자가 알지 못한 채
데이터가 소진되는 상황은 받아들일 수 없다.

첫 실행 시 다음을 명시적으로 묻는다.

```
┌─ 백그라운드 자동 업데이트 ─────────────────────────────┐
│                                                        │
│  원격 저장소의 변경사항을 미리 받아두면 Fetch 버튼을     │
│  눌렀을 때 기다리지 않고 즉시 결과를 볼 수 있습니다.     │
│                                                        │
│  다만 백그라운드에서 네트워크를 사용하므로, 실제로       │
│  보지 않을 데이터까지 내려받을 수 있습니다.             │
│                                                        │
│  ⓘ 이 설정은 언제든 [설정 > 네트워크]에서 변경할 수      │
│     있으며, 사용량 통계도 함께 확인할 수 있습니다.       │
│                                                        │
│        [ 사용 안 함 (기본) ]   [ 사용 ]                 │
└────────────────────────────────────────────────────────┘
```

**설계 규칙**

1. 질문을 미루거나 창을 닫으면 **비활성 상태로 둔다.** 무응답을 동의로 해석하지 않는다.
2. 활성화한 경우에도 §5.2의 배터리·종량제 회선 조건에서는 자동 중단한다.
3. 설정 화면에 prefetch가 실제로 절감한 대기시간과 소비한 트래픽을 **함께** 보여준다.
   사용자가 이득과 비용을 같은 화면에서 비교해 판단할 수 있어야 한다.

> 🔶 실측 결과 prefetch가 소비하는 추가 트래픽이 미미한 것으로 확인되면
> (즉, 어차피 받았을 데이터를 앞당겨 받을 뿐이라면) Phase 4 이후
> 기본값을 활성화로 전환하는 것을 재검토한다. 판단 근거는
> [performance.md](performance.md) §8의 계측 데이터로 삼는다.

## 6. 통신 속도 최적화 (요약)

요구사항 G3의 핵심. 상세 설계는 [performance.md](performance.md)에 분리했다.
여기서는 적용할 기법의 목록만 정리한다.

**§1.4의 환경(트래픽 제약 / 컴퓨팅 여유 / 오프라인 잦음)을 반영한 우선순위다.**
"전송 바이트를 줄이는 기법"을 최상위에 두고, "체감 속도만 개선하고 트래픽은
늘릴 수 있는 기법"은 신중히 다룬다.

**A. 전송 바이트를 실제로 줄이는 기법 — 최우선 적용**

| 기법 | 효과 | 적용 지점 |
|------|------|-----------|
| protocol v2 강제 | ref 광고량 대폭 감소 | 모든 네트워크 명령 |
| 압축 수준 최대 고정 | CPU를 써서 전송량 절감 | 모든 네트워크 명령 |
| ref 요청 범위 축소 (`--no-tags` 등) | 불필요한 ref 전송 제거 | fetch 계열 |
| 협상 알고리즘 튜닝 | 왕복 횟수 감소 | RTT 높을 때 자동 |
| 중복 fetch 억제 | 같은 것을 두 번 받지 않음 | 앱 레벨 요청 병합 |

**B. 왕복·연결 비용을 줄이는 기법 — 적용**

| 기법 | 효과 | 적용 지점 |
|------|------|-----------|
| SSH ControlMaster 멀티플렉싱 | 연결 핸드셰이크 제거 | SSH 원격 |
| HTTP keep-alive / 커넥션 재사용 | TLS 핸드셰이크 제거 | HTTPS 원격 |
| 다중 원격 병렬 fetch | 벽시계 시간 단축 | 원격 2개 이상 (동시 4개 상한) |

**C. 로컬 처리 가속 — 적극 적용 (컴퓨팅이 여유롭다)**

| 기법 | 효과 | 적용 지점 |
|------|------|-----------|
| commit-graph (`--changed-paths`) | 순회 가속은 **실측 오차 수준**(§4.1.1.3). 경로 조회(`git log -- path`) 가속은 미측정 🔶 | 경로 히스토리 기능 도입 시, 벤치 확인 후 |
| core.fsmonitor / untrackedCache | status 소요시간 대폭 단축 | 저장소 열 때 |
| ~~커밋 그래프 레이아웃 캐시~~ | **철회 (ADR-10)** — G2 실측 충족으로 근거 소멸 | - |

**D. 조건부 적용 — 기본 비활성** 🔶

| 기법 | 왜 기본이 아닌가 | 정책 |
|------|-----------------|------|
| partial clone (`--filter`) | 누적 트래픽을 늘릴 수 있고 오프라인 작업과 충돌 (§2.4) | clone 다이얼로그에서 선택 |
| shallow clone + deepen | 기능 제약이 크고 결국 전체를 받게 됨 | 사용자 선택 |
| sparse-checkout | 전송량이 아닌 디스크·checkout 최적화 | 모노레포에서 선택 |
| 백그라운드 prefetch | **트래픽을 소비하는 기능.** 안 쓸 데이터도 받을 수 있음 | **첫 실행 시 사용자에게 질문** (§5.4) |

**전제**: 서버는 수정하지 않는다. 모든 기법은 GitHub / GitLab / Bitbucket 등
표준 Git 서버에 그대로 적용 가능한 것만 채택했다.

## 7. 오류 처리 전략

| 계층 | 처리 방식 |
|------|-----------|
| Infrastructure | pygit2/git CLI의 오류를 `GitClientError` 계열 예외로 변환. 원문은 `detail`에 보존. raw 예외가 이 층 밖으로 새어나가면 안 된다. |
| Application | 유스케이스가 복구 가능 여부를 판단. 재시도/롤백 결정. (Phase 2+) |
| ViewModel | 데이터 표시 전담 (QAbstractItemModel). 오류 매핑 책임 없음 — 초안이 여기 두었던 "사용자 메시지 매핑"은 표현 계층으로 이동. |
| UI (표현) | 예외를 사용자 표시용 메시지 + 권장 조치로 렌더링(`_report`). 심각도별 채널 구분(토스트/다이얼로그/상태바)은 Phase 2+ — Phase 1 읽기 전용의 실패는 전부 하드 실패라 단일 모달이 정확하다. |

**Phase 1 최소 구현의 범위** (의도된 축소 — ADR-13)

- 오류 메시지의 한국어 문구를 Infrastructure의 raise 지점에서 작성한다.
  소규모라 층 분리 비용이 이득을 넘기 때문이며, Phase 5 i18n에서 정리한다.
- 권장 조치는 `GitClientError.action` 필드 하나로 나른다. 심각도 분류 체계는
  실제 케이스(index.lock, non-fast-forward, 네트워크 단절)가 등장하는
  Phase 2/3에 대고 설계한다 — 지금 만들면 채울 케이스가 없다.
- 커밋이 아닌 대상을 가리키는 태그나 dangling 참조는 목록에서 제외하되
  **반드시 로그를 남긴다.** 조용한 스킵은 디버깅 불가능한 동작이다.

**특별히 다뤄야 할 상황** (Phase 2/3에서 구체화)

- 네트워크 단절 중 push → 자동 재시도 없음. 명시적 재시도 버튼 제공.
- 인덱스 잠금(`index.lock`) 잔존 → 다른 프로세스 확인 후 해제 안내.
- 병합 충돌 → 오류가 아니라 **정상 상태 전이**로 다룬다. 충돌 해결 모드로 진입.
- 원격 거부(non-fast-forward) → pull 후 재시도, rebase, force push 세 선택지를
  각각의 결과와 함께 제시.

## 10. 개발 로드맵

### Phase 1 — 기반 (읽기 전용) ✅ 완료
- 저장소 열기, 커밋 그래프 렌더링, diff 뷰, 브랜치/태그 목록
- **완료 기준**: 어떤 저장소든 열어서 히스토리를 탐색할 수 있다. ✅
  G2도 실측으로 충족(§4.1.1.2). 정합성 감사에서 드러난 G4 예산 초과
  (refs 5천 개 → 3.1s, 대형 diff → 454ms)는 RefsLoader/DiffLoader로 해소했다.
- **다른 Phase로 이월**:
  - 단어 단위 diff(라인 내 강조), 구문 강조, 나란히 보기(side-by-side) → Phase 5
  - (선택) 상태바 ahead/behind → Phase 3 인접 시점
  - (레이아웃 캐시는 불필요로 판명되어 철회 — ADR-10)

### Phase 2 — 로컬 쓰기 ✅ 완료 기준 충족
- stage/unstage(헝크 단위 포함), commit, amend, branch 생성/삭제/전환, stash
- **완료 기준**: 이 클라이언트만으로 로컬 개발 사이클이 돌아간다. ✅

**증분 1**: 파일 단위 stage/unstage/discard, commit/amend(HEAD 메시지
프리필), 브랜치 생성·전환·삭제(컨텍스트 메뉴, 파괴적 작업 확인), stash
보관/꺼내기, 작업 디렉터리 패널과 스테이징/미스테이징 diff 보기.
쓰기는 전부 `WriteQueue`로 직렬화된다(§3.3 규칙 3 최초 실구현).

**증분 2**: 헝크/라인 단위 부분 스테이징(§4.2.1). 패치 합성기는 순수
파이썬(`domain/patch.py`)이며, diff 뷰에서 여러 줄을 선택해 그 줄만
올리거나 내릴 수 있다. 선택한 줄은 화면에서 시각적으로 구분된다 —
무엇을 고르는지 보이지 않으면 부분 스테이징은 쓸 수 없다.

**Phase 2 범위에서 제외한 것** (다른 Phase로 이월)

- 파일시스템 감시(§4.3) — 외부 CLI 조작이 즉시 반영되지 않는다(F5로 수동 갱신).
  워커/디바운싱 설계가 필요해 별도 작업으로 분리한다.
- 커밋 후 그래프 증분 갱신 — 현재는 전체 재로딩(수백 ms, G2 예산 내)이라
  스크롤 위치가 초기화된다. 체감 개선이지 기능 결손은 아니다.
- 충돌(CONFLICTED) 상태의 해결 UI — 표시는 되나 조작이 없다.
  머지를 만드는 쪽이 Phase 4이므로 그쪽에서 함께 다룬다.

### Phase 3 — 원격 통신 🚧 증분 1 (계측 체계) 구현 완료
- clone, fetch, pull, push, 원격 관리, 인증 연동
- **계측 체계를 가장 먼저 구축한다** ([performance.md](performance.md) §8).
  이후 §9의 1~6순위 최적화를 적용한다.
- 🔶 표시된 잠정 결정(전체 clone 기본값, prefetch 옵트인)을 실측 데이터로 검증하고,
  결과가 다르면 문서와 함께 결정을 갱신한다.
- **완료 기준**: 원격 협업이 가능하고, 누적 전송량 절감 효과를 실측으로 제시한다.

**증분 1 (구현됨)**: git CLI 엔진, Trace2 기반 계측, SQLite 이력 저장, fetch.
자세한 것은 §4.6. 이제 "얼마나 줄었는가"를 숫자로 답할 수 있다 —
이후 최적화 판단의 근거가 여기서 나온다.

**증분 2 (구현됨)**: push, pull(빨리 감기까지), ahead/behind 표시.
자세한 것은 §4.7. 계측이 있었기에 ADR-8(압축 최대 고정)이 근거 없는 설정임을
숫자로 확인하고 정정할 수 있었다 — 계측을 먼저 만든 값을 처음 회수한 지점이다.

**증분 3 (남음)**: clone(필터 선택 UI 포함), 원격 관리(추가·삭제·이름 변경),
인증 UI(credential helper 프롬프트 중계), §9의 나머지 최적화
(SSH 멀티플렉싱, 협상 알고리즘 튜닝, 중복 요청 억제),
`pack.window`/`pack.depth`의 실제 효과 측정.

### Phase 4 — 고급 작업
- merge / rebase(인터랙티브 포함) / cherry-pick / revert / reset
- 충돌 해결 UI, reflog 탐색
- **완료 기준**: GitKraken의 핵심 기능 집합을 커버한다.

### Phase 5 — 다듬기
- 테마, 단축키, 설정 화면, 성능 튜닝, 다국어

| 7 | **prefetch 기본 비활성, 첫 실행 시 질문** | 기본 활성 / 기능 제외 | 체감 개선 효과는 크지만 사용자가 요청하지 않은 트래픽을 소비함. 동의 없이 켜지 않는다. (§5.4) | 🔶 잠정 — Phase 4 이후 재검토 |
| 8 | ~~**압축 수준 최대 고정**~~ → **압축 하한선 방어** | 회선 상태별 자동 조정 / 설정 제거 | ~~CPU가 남고 트래픽이 비싸므로 항상 최대 압축이 유리~~ → **실측으로 정정.** 기본값(6) 대비 최대(9)는 전송 바이트 **0.00%** 차이(66,591 동일), 소요시간도 동일. 목적함수가 옳아도 한계 이득이 0이면 의미가 없다. 설정은 유지하되 목적이 바뀌었다 — 사용자가 `compression=0`으로 꺼둔 경우(2.1배 증가)를 막는 하한선. ([performance.md](performance.md) §6.4.1) | 정정 — Phase 3 실측 |
| 9 | **커밋 로딩을 밀어넣기(push) 방식으로** | 뷰포트 기반 지연 로딩(pull) | 정렬된 순회는 비용의 83%를 첫 결과 이전에 치르므로 지연 로딩이 성립하지 않음. UI 스레드 밖으로 옮김. (§4.1.1.1) | 확정 — 실측 근거 |
| 10 | **레이아웃 캐시(SQLite) 도입하지 않음** | 캐시 구현 | 도입 근거였던 G2 미달이 벤치마크 아티팩트로 판명됨. 실측 493ms로 목표를 3배 여유로 충족. 근거 없는 복잡도를 만들지 않는다. (§4.1.1.2, §4.1.1.3) | 확정 — 철회 |
| 11 | **벤치마크 저장소는 팩된 상태로 생성** | pygit2 루프로 생성 | 느슨한 오브젝트 저장소는 순회가 40배 느려 현실을 대표하지 못함. `git fast-import` 사용. (§4.1.1.3) | 확정 — 재발 방지 |
| 12 | **설정·앱 상태는 QSettings, SQLite는 캐시 전용** | 단일 SQLite 스토어 | Phase 1 영속화 수요는 문자열 하나(마지막 저장소 경로). 이를 위한 SQLite 스토어는 ADR-10이 배격한 근거 없는 복잡도. 구현이 먼저 옳게 갔고 문서를 맞춤. (§4.5) | 확정 |
| 13 | **오류 매핑은 표현 계층, Phase 1은 단일 모달 + action 필드** | ViewModel 매핑 계층 + 심각도 라우팅 | ViewModel(QAbstractItemModel)은 데이터 표시 전담. 심각도 분류는 실제 케이스가 등장하는 Phase 2/3에 설계 — 지금 만들면 채울 케이스가 없음. (§7) | 확정 — Phase 2/3 확장 예정 |
| 14 | **commit-graph 자동 생성 안 함** | 저장소 열 때 자동 생성 | 순회 가속이 실측 오차 수준(4.9→4.7µs, §4.1.1.3). `--changed-paths` 경로 조회 이득만 잠정 후보로 유지, 소비 기능 등장 시 벤치 후 결정. | 🔶 잠정 |
| 15 | **G4를 "블로킹 0ms"에서 "단일 블록 ≤ 50ms" 예산으로** | 절대 금지 유지 | 0ms는 측정 불가능하고 이미 수용된 고정비(핸들 열기 ~20ms)와 모순. 예산은 검증 가능. (§3.3) | 확정 |
| 16 | **원격 명령에 `transfer.unpackLimit=1` 고정** | git 기본값(100) | 기본값에서는 작은 fetch가 팩을 풀어 `Receiving objects` 줄이 사라져 **전송량 측정 자체가 불가능**하다. 목적함수가 누적 바이트인데 측정이 비면 최적화를 판정할 수 없다. 저장소가 팩된 상태로 유지되는 부수효과는 ADR-11의 40배 측정과 같은 방향. (§4.6) | 🔶 잠정 — 팩 누적 임계를 측정 후 repack 정책 결정 |
| 17 | **fetch는 WriteQueue를 거치지 않음** | 모든 원격 작업 직렬화 | 쓰기 직렬화는 인덱스/워킹트리 보호가 목적인데 fetch는 원격 추적 ref만 건드리고 git이 ref 갱신을 원자적으로 처리한다. 큐에 넣으면 로컬 작업만 막힌다. (§3.3, §4.6) | 확정 |
| 18 | **원격 작업 계측을 기능보다 먼저 구현** | fetch/pull/push 먼저, 계측 나중 | §4.1.1.3에서 측정 없는 성능 결론이 이미 한 번 틀렸다. 목적함수가 전송량이므로 계측이 없으면 이후 모든 최적화가 주장에 그친다. | 확정 |
| 19 | **"팩 0바이트"와 "측정 실패"를 분리** | 둘 다 미측정으로 기록 | 변경 없는 fetch가 일상의 대다수라, 미측정으로 세면 "측정 공백" 신호가 상시 오탐이 되어 진짜 공백을 가린다. `remote: Total` 줄의 유무가 판별자. (§4.6.1) | 확정 — 리뷰 실측 |
| 20 | **`--no-recurse-submodules` 고정** | git 기본값(on-demand) 유지 | 재귀 fetch는 서브모듈 바이트를 기록하지 않으면서 상위 저장소 참조 수만 부풀린다(실측: 70KB 전송이 329B로 기록). "측정 성공, 값만 틀림"이라 집계표에서 탐지되지 않는다. 계측의 귀속 단위는 저장소 하나여야 한다. | 🔶 잠정 — 서브모듈 지원 시 저장소별 행으로 재설계 |
| 21 | **원격 명령을 비대화형으로 강제하고 타임아웃 시 프로세스 트리를 종료** | `GIT_TERMINAL_PROMPT=0`만으로 충분하다고 가정 | 그 변수는 git 자체 프롬프트만 끈다. credential helper의 GUI와 askpass 체인이 워커를 무기한 붙잡고, 파이프를 상속한 자손 때문에 타임아웃도 상한 노릇을 못 한다. 둘 다 실측 확인. (§4.6.2) | 확정 |
| 22 | **pull을 fetch(CLI) + 빨리 감기(pygit2)로 쪼갠다** | `git pull` 한 번에 위임 | ADR-2의 경계를 지킨다 — 전송은 CLI, 로컬 쓰기는 pygit2. 덕분에 전송 절반을 그대로 계측하고, 쓰기 절반만 WriteQueue에 태워 §3.3 규칙 3을 지킬 수 있다. `git pull`에 맡기면 계측도 직렬화도 불투명해진다. (§4.7) | 확정 |
| 23 | **빨리 감기는 checkout 먼저, 참조 이동 나중** | 참조를 먼저 옮기고 checkout / `reset --hard` | 순서를 바꾸면 libgit2가 이미 같아진 HEAD와 비교해 **워킹트리를 갱신하지 않는다 — 오류 없이.** 수정·삭제가 반영되지 않고 인덱스에 엉뚱한 변경이 남는다(실측). `reset --hard`는 동작하지만 커밋하지 않은 작업을 조용히 덮어쓴다. 이 순서는 더러운 트리에서 checkout이 거부되어 **실패해도 안전하다.** (§4.7.1) | 확정 — 실측 근거 |
| 24 | **병합이 필요한 pull은 시작하지 않고 안내한다** | 자동 병합 커밋 생성 | 충돌 해결 UI가 Phase 4다. 지금 병합을 시작하면 사용자가 앱 안에서 끝낼 수 없는 상태(충돌 인덱스)에 갇힌다. 받아온 객체는 버리지 않는다 — 트래픽을 이미 썼으므로 다시 받게 하면 안 된다. (§4.7.2) | 🔶 잠정 — Phase 4에서 해제 |
| 25 | **upstream은 설정에서 읽는다. 규약으로 추측하지 않는다** | `<origin>/<브랜치명>` 조합 | 추측하면 fork 워크플로에서 엉뚱한 ref를 대상으로 삼고 "이미 최신"이라 답하고, 분리된 HEAD에서는 `refs/remotes/<remote>/HEAD`가 만들어져 사용자의 HEAD가 원격 기본 브랜치로 끌려간다. 둘 다 실측 확인. (§4.7.3) | 확정 |
| 26 | **실패한 원격 작업의 계측도 기록한다** | 성공한 작업만 기록 | push는 ref 단위로 원자적이라 부분 성공이 실재한다(exit 1인데 일부는 전송됨). 버리면 "측정 실패"로도 안 잡히는 완전 무기록이 되어 누적 바이트가 조용히 줄어든다. 단 실패 경로에서는 0 채우기를 하지 않는다. (§4.7.5) | 확정 |
| 27 | **WriteQueue의 idle은 방출 이후 상태를 다시 읽고 낸다** | `_pending` 스냅샷으로 판정 | 작업 완료 시그널이 직접 연결이라, 슬롯이 모달을 띄우면 그 중첩 루프에서 새 작업이 제출·시작될 수 있다. 그 상태에서 idle을 내면 워킹트리를 쓰는 중에 UI가 전체 재로딩을 시작한다. 큐가 자기 불변식을 지키게 한다. | 확정 |
