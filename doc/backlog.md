## 0. 이 문서를 읽는 법

**흩어져 있던 남은 일을 한곳에 모은 목록이다.** 이전에는 design.md §13(감사가
찾아낸 결함), §10(Phase 로드맵), performance.md §9(최적화 우선순위), README
("아직 없는 것")에 나뉘어 있어 "다음에 뭘 하지"에 답하려면 네 곳을 봐야 했다.

**우선순위의 근거는 목적 함수다.** 이 앱이 최소화하는 것은 누적 전송 바이트가
아니라 **사용자가 네트워크를 기다리는 시간**이다 (ADR-56, design.md §1.4).
요금제 제약은 없고 회선이 느릴 뿐이다. 그래서:

- CPU 시간도 대기 시간이다 — `절약한 전송 시간 > 추가 CPU 시간`일 때만 이득
- 누적 총량 관리는 폐기됐다 (ADR-57)
- 임계 경로 **밖**의 전송은 총량이 늘어도 이득이다 (ADR-7)

**상태 표기**

| 표기 | 뜻 |
|------|-----|
| 🔴 | 사용자에게 잘못된 결과나 손실을 줄 수 있다 |
| 🟡 | 기능 공백 — 앱 밖으로 나가야 하는 일이 남아 있다 |
| 🟢 | 개선 여지 — 지금도 동작한다 |
| ⚪ | 결정 대기 — 무엇을 할지부터 정해야 한다 |

## 2. 기능 공백

### 2.1 rebase / cherry-pick / revert / reset (Phase 4 증분 3)

🟡 **가장 큰 공백이다.**

증분 2에서 충돌 해결을 만들면서 rebase·cherry-pick 충돌은 **일부러 감췄다**
(ADR-65). 두 가지 때문이다:

1. **스테이지 2/3의 의미가 병합과 반대다.** rebase는 upstream을 체크아웃한
   뒤 사용자 커밋을 재생하므로 stage 2가 "올라탈 곳", stage 3이 "사용자 자신의
   커밋"이다. 병합 기준 라벨을 그대로 두면 **"내 것 사용"이 사용자의 커밋을
   버린다** — 실측에서 `rebase --continue`가 성공으로 끝나며 커밋이 조용히
   사라졌다.
2. **끝낼 방법이 없다.** 앱에 `--continue`가 없어 해결해 놓고 마무리를 못 한다.

**할 일**
- 저장소 상태에 따라 충돌 양쪽의 이름을 정한다 (`is_rebasing()` 등)
- `--continue` / `--skip` / `--abort` 경로
- `index_conflicts()`의 상태 제한을 푼다 (design.md §4.10.10)
- reset은 파괴적이라 §5.2 확인 절차가 필요하다

**참고**: design.md §4.10.10, ADR-65

### 2.2 원격 관리 (Phase 3 증분 5)

🟡 원격 추가·삭제·주소 변경이 없다. 지금은 git CLI로 나가야 한다.

`RemoteEngine.list_remotes()`와 `remote_url()`은 이미 있다.

### 2.3 줄 단위로 골라 합치기

🟢 지금은 파일 단위 선택(내 것/상대 것)과 편집기로 마커를 정리하는 경로 둘이다.
양쪽에서 일부씩 가져오려면 편집기를 써야 한다.

부분 스테이징(§4.2)의 패치 합성 기계를 재사용할 수 있을 것으로 보인다.

### 2.4 파일시스템 감시

🟢 외부 CLI 조작이 화면에 자동 반영되지 않는다 — F5로 수동 갱신한다.
`watchdog` 도입 시 §3.3 스레딩 규칙과 디바운스 설계가 필요하다.

**참고**: design.md §4.5

## 3. 정확성 결함 (감사에서 확정됨)

> design.md §13에서 옮겨왔다. **적어두지 않으면 "감사를 통과했다"로 읽힌다.**

### 3.1 진행 중인 원격 작업을 멈출 수 없다

🔴 취소 경로는 구현돼 있고(저장소 전환·창 닫기가 쓴다) **UI에 버튼만 없다.**
느린 회선에서 대기가 목적 함수인데 빠져나갈 길이 없다.

진행 표시(§4.6.5) 위에 얹는 작업이라 규모가 작다.

### 3.2 `compression=9`는 "하한선"이 아니라 최대값이다

🔴 실측상 기본값(6) 대비 바이트는 **0.00~0.5%**만 줄면서 CPU를 크게 쓴다.
ADR-56 이후 CPU 시간도 대기 시간이므로, 하한선이 목적이라면 값은 **6**이어야
한다. 지금 설정은 목적 함수를 거스른다.

값을 바꾸면 전 원격 경로의 계측 기준선이 이동하므로 재측정과 함께 다뤄야 한다.

**참고**: performance.md §6.4, ADR-8

### 3.3 ahead/behind가 UI 스레드를 벌어진 만큼 쓴다

🟢 평소엔 무시할 수준이지만 크게 벌어지면 G4 예산(50ms)에 닿는다.
워커로 옮기는 것 자체는 간단하나 상태바 갱신 흐름 전체와 함께 봐야 한다.

### 3.4 부분 복제에서 없는 객체를 만났을 때의 처리가 없다

⚪ `_translate`가 OID를 버리고 일반 오류로 뭉개, 사용자는 무엇이 없는지도
어떻게 받는지도 알 수 없다. 부분 복제는 선택지로만 제공되고 기본값이 아니라
(ADR-6 확정) 급하지 않다.

### 3.5 실패한 clone의 계측이 남지 않는다

⚪ ADR-26("실패해도 쓴 트래픽은 기록한다")과 §4.9("귀속시킬 대상이 없다")가
서로 다른 말을 한다. **저장소가 없는 작업의 계측을 어디에 귀속시킬지**부터
정해야 한다.

## 4. 목적 함수 최적화 (미구현)

> performance.md §9 우선순위표에서 🔶로 표시된 것들.

| 항목 | 기대 효과 | 상태 |
|------|----------|------|
| SSH ControlMaster 멀티플렉싱 | 연결 핸드셰이크 제거 — **왕복이 곧 대기다** | 🟢 미구현 |
| 협상 알고리즘 튜닝 | 왕복 횟수 감소 (RTT 높을 때) | 🟢 미구현 |
| 중복 fetch 억제 | 같은 것을 두 번 받지 않음 (앱 레벨 요청 병합) | 🟢 미구현 |
| 유휴 시 로컬 repack | **델타 설정(ADR-35)을 쓸 유일한 자리** — 사용자가 기다리지 않는 시간의 CPU는 공짜다 | 🟢 미구현 |
| core.fsmonitor / untrackedCache | status 소요시간 단축 | 🟢 미구현 |
| commit-graph (`--changed-paths`) | 경로 조회 가속 (순회 가속은 실측 오차 수준) | ⚪ 미측정 |

**정정된 목적 함수에서 앞의 셋이 올라갔다.** 왕복 감소는 곧 대기 감소이고,
느린 회선일수록 RTT의 비중이 크다.

`transfer.unpackLimit=1`이 만드는 팩 누적은 원격 작업 뒤 `maintenance run --auto`로
치우고 있다(design.md §4.6.2). 유휴 repack은 그것과 별개로 델타 설정을 시험할
자리다.

## 7. 다시 논의하지 않기로 한 것

**폐기된 결정을 다시 꺼내지 않으려고 적어둔다.** 근거가 무너진 것들이다.

| 항목 | 왜 안 하는가 |
|------|-------------|
| 월간 트래픽 예산 · 상한 · 누적 대시보드 | **요금제 제약이 없다** (ADR-57). 청구서가 없는데 예산을 관리할 이유가 없고, 상한은 필요한 순간에 작업을 막는다 |
| prefetch 온보딩 질문 | 물어볼 비용이 없다 (ADR-7). 대신 `저장소 > 배경에서 미리 가져오기`로 끌 수 있다 |
| 압축 수준 최대화 | 기본값(6) 대비 **0.00%** (ADR-8 정정). §3.2 참고 |
| `pack.window`/`pack.depth` 델타 설정 (push) | 바이트 −0.5~4%에 소요시간 **+20%** (ADR-35). 유휴 repack에서만 쓸 값이다 |
| 남은 시간(ETA) 표시 | 역산이 구조적으로 낙관 편향 (ADR-59). 팩이 커밋→트리→blob 순으로 쓰여 객체당 바이트가 뒤로 갈수록 커진다 |

## 8. 이 문서를 갱신하는 법

- 항목을 **끝내면 지운다.** 완료 목록이 아니라 남은 일 목록이다.
- 감사나 리뷰에서 새 결함이 확정되면 §3에 넣는다 — 고치지 않을 것이라면
  **왜 안 고치는지**를 함께 적는다. 이유 없이 남은 항목은 다음 사람이
  판단할 수 없다.

- design.md의 §13은 이 문서로 대체됐다.
