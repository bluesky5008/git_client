## 1. 문제 정의

"Git 클라이언트-서버 통신 속도 최대화"를 달성하려면 먼저 **무엇이 느린지**를
정확히 나눠야 한다. 체감 대기시간은 다음 네 가지의 합이다.

```
총 대기시간 = 협상(negotiation) + 서버 팩 생성 + 전송 + 로컬 적용
```

| 구간 | 주 원인 | 클라이언트가 통제 가능한가 |
|------|---------|---------------------------|
| 협상 | ref 광고량, 왕복 횟수(RTT × N) | **가능** — protocol v2, 협상 알고리즘 |
| 서버 팩 생성 | 서버 CPU, 델타 압축 | 부분적 — 요청 범위를 줄이면 감소 |
| 전송 | 대역폭, 전송 바이트 수 | **가능** — partial clone, shallow |
| 로컬 적용 | 디스크 I/O, 인덱스 갱신, checkout | **가능** — sparse-checkout, 인덱스 튜닝 |

### 1.1 최적화 목적함수

최적화에는 방향이 필요하다. 무엇을 최소화할 것인가?

이 프로젝트의 목적함수는 **사용자가 네트워크를 기다리는 시간이다.**

```
minimize  Σ(사용자가 기다리는 시간)   ← 1순위
  이때 전송 바이트와 왕복 횟수가 주된 지렛대다 (회선이 느리므로)
subject to  UI 응답성 유지            ← 제약조건 (G2)
            오프라인 작업 가능         ← 제약조건
사용 가능 예산: CPU, 메모리, 디스크 — 단 **CPU 시간도 대기 시간이다**
```

> **정정 (v0.7).** 초기 문서는 목적함수를 "누적 전송 바이트"로 두고, 그
> 근거를 "트래픽 총량이 제한된 자원"에서 찾았다. **실제 환경에는 요금제
> 제약이 없다** — 문제는 순전히 속도다 (design.md §1.4 정정).
>
> 대부분의 결론은 그대로다. 느린 회선에서 바이트는 곧 시간이므로 바이트를
> 줄이는 최적화는 여전히 옳다. 하지만 **세 가지가 달라진다**:
>
> 1. **CPU와 전송이 같은 화폐가 된다.** 예전에는 "CPU는 공짜"였으므로 바이트를
>    조금이라도 줄이면 이득이었다. 지금은 CPU 시간도 사용자가 기다리는 시간이라
>    `절약한 전송 시간 > 추가 CPU 시간`일 때만 이득이다. 델타 설정(§6.4.3,
>    −0.5~4% 바이트 / +20% 시간)이 이 기준에서 명확히 탈락한다.
> 2. **누적 총량은 관리 대상이 아니다.** 청구서가 없다. 월간 예산·상한·
>    대시보드는 삭제한다(§5.2, §8.4). 관리할 것은 **한 번의 대기 길이**다.
> 3. **임계 경로 밖의 전송은 거의 공짜다.** prefetch처럼 사용자가 기다리지
>    않는 시간에 받아두는 것은 총 전송량이 늘어도 이득이다.

[design.md §1.4](design.md)의 환경 — **회선은 느리고 컴퓨팅은 남는다** —
가 이 목적함수를 결정한다. 일반적인 Git 튜닝 조언은 빠른 회선을 가정하므로
일부는 여기서 **반대로 적용해야 한다** (예: "빠른 회선에서는 압축을 낮춰라").

**세 가지 파생 원칙**

1. **전송량을 줄이는 것이 대역폭을 늘리는 것보다 항상 효과적이다.**
   회선 속도는 바꿀 수 없지만 받아야 할 바이트 수는 바꿀 수 있다.
   설계의 중심은 "빠르게 받기"가 아니라 **"안 받아도 되는 것을 안 받기"**다.

2. **CPU와 전송은 같은 화폐로 계산한다 — 둘 다 사용자의 시간이다.**
   `절약한 전송 시간 > 추가로 쓴 CPU 시간` 일 때만 채택한다.

   이 기준으로 셋이 탈락했다 — 압축 최대화는 기본값 대비 **0.00%**(§6.4.1),
   델타 설정은 0.5~4% 절감에 소요시간 **+20%**(§6.4.3), commit-graph는 오차
   수준(§6.1). 특히 델타 설정은 옛 기준("바이트를 줄였으니 이득")으로는
   채택할 뻔했다.

   예외는 **사용자가 기다리지 않는 시간에 쓰는 CPU**다 — 유휴 시 repack처럼.
   그때는 CPU가 실제로 공짜다.

3. **임계 경로 밖으로 옮긴 전송은 총량이 늘어도 이득이다.**
   요금이 없으므로 "쓸모없을 수도 있는 데이터를 미리 받는" 비용은 회선
   점유와 배터리뿐이다. prefetch가 여기 해당한다.

4. ~~**체감 속도 개선이 트래픽을 늘린다면 신중해야 한다.**~~ (v0.7에서 폐기 —
   근거였던 트래픽 비용이 존재하지 않는다)
   prefetch는 대기시간을 0에 가깝게 만들지만, 사용자가 결국 보지 않을
   데이터까지 받을 수 있다. 소요시간만 보면 명백한 이득이지만
   목적함수가 바이트라면 **자동으로 정답이 아니다.** (§5 참조)

세 번째 원칙이 이번 개정의 핵심 변경 사항이다. 초안은 소요시간 최소화를
암묵적으로 가정하고 prefetch와 partial clone을 기본값으로 삼았으나,
목적함수가 바뀌면서 두 결정 모두 뒤집혔다.

### 1.2 전제

- **서버는 수정하지 않는다.** GitHub / GitLab / Bitbucket / Gitea 등
  표준 Git 서버에 그대로 통하는 기법만 채택한다.
- 모든 네트워크 작업은 git CLI를 경유한다 ([design.md](design.md) §2.3 참조).
- 서버가 특정 기능을 지원하지 않을 수 있으므로 **capability 조회 후 자동 폴백**이 필수다.
- 컴퓨팅 자원은 제약이 아니다. CPU·디스크를 쓰는 최적화에 인색할 이유가 없다.
- 🔶 표시는 실측 데이터 없이 내린 잠정 결정이며 Phase 3 이후 재검토 대상이다.

## 2. 협상 구간 최적화

### 2.1 protocol v2

**문제**: protocol v0에서 서버는 연결 즉시 **모든 ref를 광고한다.**
ref가 5만 개인 저장소라면, 클라이언트가 브랜치 하나만 원해도
수 MB의 ref 목록을 먼저 받아야 한다.

**해결**: protocol v2는 클라이언트가 필요한 ref만 필터링해 요청할 수 있다.

```bash
git -c protocol.version=2 fetch origin main
# 서버에 "refs/heads/main 관련만 달라"고 요청 → 광고량이 수십 KB로 축소
```

**적용 방식**

- git 2.26+ 부터 fetch의 기본값이 v2지만, 사용자 설정이나 오래된 git이
  이를 덮을 수 있다. 클라이언트는 **모든 네트워크 명령에 `-c protocol.version=2`를
  명시적으로 부여한다.** 전역 설정을 건드리지 않고 명령 단위로 강제하는 방식이라
  사용자의 기존 환경을 오염시키지 않는다.
- 서버가 v2를 지원하지 않으면 git이 자동으로 v0로 폴백하므로 안전하다.

### 2.2 ref 요청 범위 축소

fetch 시 필요한 refspec만 지정한다.

```bash
# 나쁨: 모든 브랜치 + 모든 태그
git fetch origin

# 좋음: 화면에 필요한 것만
git fetch --no-tags origin refs/heads/main:refs/remotes/origin/main
```

- 백그라운드 prefetch는 `--no-tags`를 기본으로 한다.
  태그는 사용자가 태그 뷰를 열 때만 별도로 가져온다.
- 브랜치가 수천 개인 저장소에서는 **최근 활동 브랜치 + 사용자가 추적 중인 브랜치**만
  우선 fetch하고, 나머지는 지연 로딩한다.

### 2.3 협상 알고리즘

클라이언트와 서버는 "내가 이미 가진 커밋"을 주고받으며 공통 조상을 찾는다.
기본 알고리즘은 히스토리가 복잡할 때 왕복이 많아진다.

```bash
git -c fetch.negotiationAlgorithm=skipping fetch origin
```

`skipping` 알고리즘은 커밋을 건너뛰며 탐색해 왕복 횟수를 줄인다.
RTT가 큰 원거리 서버(예: 국내 → 미국 리전, RTT 150~200ms)에서 효과가 크다.

**적용 정책** 🔶 (미구현): 측정된 RTT가 80ms를 넘으면 `skipping`을 자동
적용한다 — **아직 RTT 측정도 알고리즘 선택도 구현되지 않았다.** 현재
`BASE_CONFIG`에는 `fetch.negotiationAlgorithm`이 없어 git 기본값을 쓴다.
계측(§8)이 왕복 횟수를 이미 기록하므로, 그 데이터가 쌓이면 임계값을
추측이 아니라 실측으로 정할 수 있다.

## 3. 전송량 최적화

### 3.1 Partial Clone — 선택지로 제공, 기본값 아님 🔶

**원리**: 커밋과 트리(디렉터리 구조)만 받고, **파일 내용(blob)은 실제로 필요할 때
받는다.** 필요 시점에 git이 자동으로 서버에서 개별 blob을 가져온다.

```bash
git clone --filter=blob:none https://github.com/org/repo.git
```

초기 전송량만 보면 이 기법의 효과는 압도적이다. 대부분의 저장소에서
히스토리 blob이 용량의 대다수를 차지하고, 그 대부분은 clone 직후에 쓰이지 않는다.

**그럼에도 기본값으로 삼지 않는 이유**

§1.1의 목적함수는 **초기 전송량이 아니라 누적 전송량**이다. 이 관점에서 보면
partial clone은 전송을 없애는 게 아니라 **미루고 잘게 쪼갠다.**

```
전체 clone 누적:    [━━━━━━━━━━ 초기 1회 ━━━━━━━━━━] + 증분 fetch
partial clone 누적: [━━ 초기 ━━] + ▪▪▪▪▪▪▪▪▪▪▪▪▪▪▪▪▪▪▪▪ (blob 요청이 계속 누적)
                                   ↑ 각 요청마다 HTTP 헤더·TLS·왕복 오버헤드
```

구체적인 손해 요인:

| 요인 | 설명 |
|------|------|
| 요청당 오버헤드 | blob을 개별 요청하면 요청마다 헤더·왕복 비용이 붙는다. 큰 팩 하나로 받는 것보다 바이트당 효율이 나쁘다. |
| 델타 압축 효율 저하 | 팩 파일은 유사한 객체들을 델타로 묶어 압축한다. 객체를 낱개로 받으면 이 이득을 잃는다. |
| 재요청 가능성 | 로컬 GC나 저장소 재생성 시 지연 로딩했던 객체를 다시 받게 될 수 있다. |
| 오프라인 충돌 | 오프라인 작업이 잦은 환경에서 과거 파일을 열람할 수 없다. 이건 성능 문제가 아니라 **기능 상실**이다. |

**따라서 기본값은 전체 clone이다.** 한 번에 크게 받고, 이후로는 증분만 받는 쪽이
트래픽 총량과 오프라인 요구를 동시에 만족한다.
([design.md §2.4](design.md) 참조)

**선택지로는 계속 제공한다**

clone 다이얼로그에서 필터 수준을 고를 수 있게 한다.

| 필터 | 의미 | 적합한 상황 |
|------|------|-------------|
| 없음 | 전체 clone | **기본값.** 오프라인 작업이 있거나 히스토리 탐색이 잦은 경우 |
| `blob:limit=<크기>` | 지정 크기 초과 blob만 지연 | 대용량 바이너리 에셋이 섞인 저장소. **절충안으로 가장 유력** |
| `blob:none` | 모든 blob 지연 | 초기 clone이 트래픽 예산을 초과하는 매우 큰 저장소 |
| `tree:0` | 트리까지 지연 | 특정 커밋만 일회성으로 확인할 때 |

**선택 시 클라이언트가 지원해야 할 것**

1. **비용 명시**: 필터를 고르면 "오프라인에서 과거 파일을 열 수 없습니다"를
   경고로 표시한다. 사용자가 나중에 이유를 모른 채 막히면 안 된다.
2. **지연 로딩 상태 표시**: blob을 네트워크로 받는 중임을 UI에 명확히 표시한다.
3. **묶음 요청**: 지연 blob이 여러 개 필요할 때 개별 요청하지 않고 한 번에 묶어
   요청해 오버헤드를 줄인다.
4. **사후 전환 제공**: partial clone으로 받았다가 오프라인 작업이 필요해지면
   `git fetch --refetch --filter=blob:none` 해제 후 나머지를 받는 경로를 안내한다.
5. **서버 미지원 시 폴백**: 서버가 `filter` capability를 광고하지 않으면
   자동으로 일반 clone으로 전환하고 그 사실을 알린다.

> 🔶 Phase 3에서 실제 사용 패턴을 측정한다. **핵심 측정 질문**: partial clone
> 사용 시 30일 누적 전송량이 전체 clone 대비 실제로 적은가?
> 히스토리 탐색 빈도가 낮다면 `blob:limit`을 기본값으로 올리는 것을 검토한다.

### 3.2 Sparse Checkout

partial clone이 **받는 양**을 줄인다면, sparse-checkout은 **디스크에 펼치는 양**을 줄인다.
모노레포에서 특히 중요하다.

```bash
git sparse-checkout init --cone
git sparse-checkout set apps/web libs/shared
```

**효과**: 워킹 트리 파일 수가 줄어 checkout, status, 브랜치 전환이 모두 빨라진다.
50만 파일 모노레포에서 관심 디렉터리만 펼치면 브랜치 전환이 수십 초 → 1초 미만이 된다.

`--cone` 모드는 디렉터리 단위 패턴만 허용해 매칭 비용이 낮다.
임의 glob을 쓰는 비-cone 모드보다 훨씬 빠르므로 반드시 cone 모드를 쓴다.

**UI 설계**: 저장소 열 때 디렉터리 트리를 보여주고 체크박스로 관심 영역을 고르게 한다.
사용자가 sparse 대상 밖의 파일을 열려고 하면 해당 디렉터리 추가를 제안한다.

### 3.3 Shallow Clone과 점진적 Deepen

```bash
git clone --depth=1 <url>       # 최신 커밋 1개만
git fetch --deepen=500          # 이후 500개씩 확장
git fetch --unshallow           # 전체 히스토리 복원
```

**전략** 🔶 (아래 3단계는 미구현): "빨리 시작하고, 필요해지면 채운다."

> **현재 상태**: 복제 시 `--depth=1` 선택지는 **있다**(§4.9). 그러나 이후의
> 점진적 deepen — 유휴 시 백그라운드 확장, 스크롤로 경계 도달 시 트리거,
> 제약 작업에서 자동 unshallow 제안 — 은 **하나도 구현되지 않았다.**
> 지금은 shallow 상태를 상태바에 표시하는 것까지만 한다.

1. 초기 clone은 `--depth=1`로 즉시 작업 가능 상태를 만든다.
2. 앱이 유휴 상태가 되면 백그라운드로 `--deepen`을 반복 실행해 히스토리를 채운다.
3. 사용자가 그래프를 아래로 스크롤해 shallow 경계에 닿으면 즉시 deepen을 트리거한다.

**주의**: shallow 저장소는 push, merge-base 계산, blame에서 제약이 있다.
클라이언트는 shallow 상태를 상태바에 명시하고, 제약이 걸리는 작업 시도 시
"히스토리를 마저 받아야 합니다"라고 안내한 뒤 자동으로 unshallow를 제안한다.

**partial clone과의 우선순위**: 둘 다 되는 서버라면 **partial clone을 우선한다.**
shallow는 히스토리 자체가 잘려 기능 제약이 크지만, partial clone은
전체 히스토리 구조를 유지하면서 blob만 지연시키므로 제약이 훨씬 적다.

### 3.4 Bundle URI (기회적 적용)

git 2.38+ 에서 도입된 기능으로, 서버가 미리 만들어둔 팩 파일을
CDN에서 받아 clone을 가속한다.

```bash
git -c transfer.bundleURI=true clone <url>
```

서버가 `bundle-uri` capability를 광고할 때만 동작한다.
현재 지원 호스팅이 제한적이므로 **감지되면 사용, 아니면 무시**하는 기회적 적용으로 다룬다.

## 4. 연결 계층 최적화

### 4.1 SSH 연결 멀티플렉싱

**문제**: SSH fetch마다 TCP 핸드셰이크 + SSH 키 교환 + 인증이 반복된다.
RTT 150ms 회선에서 이 왕복만 400~600ms다. **fetch 자체보다 연결 수립이 더 오래 걸린다.**

**해결**: OpenSSH의 `ControlMaster`로 첫 연결을 유지하고 이후 연결이 이를 재사용한다.

클라이언트는 **사용자의 `~/.ssh/config`를 건드리지 않고**, 앱이 실행하는
git 명령에만 전용 SSH 설정을 주입한다.

```bash
GIT_SSH_COMMAND="ssh -o ControlMaster=auto \
                     -o ControlPath=<앱캐시경로>/ssh-%r@%h:%p \
                     -o ControlPersist=10m" \
  git fetch origin
```

**효과**: 두 번째 이후 연결에서 핸드셰이크 비용이 사실상 사라진다.
주기적 prefetch가 잦을수록 누적 이득이 크다.

**플랫폼 주의**: Windows 기본 OpenSSH는 ControlMaster를 지원하지 않는다.
클라이언트는 시작 시 이를 감지해, 미지원이면 HTTPS 원격 사용을 권장하거나
해당 최적화를 건너뛴다.

### 4.2 HTTP/2 및 커넥션 재사용

```bash
git -c http.version=HTTP/2 fetch origin
```

HTTP/2의 멀티플렉싱으로 요청 병렬화 이득이 있다.
단, 일부 프록시 환경에서 HTTP/2가 문제를 일으키므로
**실패 시 HTTP/1.1로 자동 재시도**하는 폴백을 넣는다.

partial clone 환경에서는 개별 blob 요청이 다수 발생하므로
커넥션 재사용의 이득이 특히 크다.

### 4.3 병렬화

| 대상 | 방법 |
|------|------|
| 다중 원격 | 원격별로 독립 프로세스를 동시에 실행 |
| 서브모듈 | `git fetch --recurse-submodules --jobs=4` |
| 다중 저장소 | 등록된 저장소들의 prefetch를 동시 실행 |

**동시 실행 상한**: 무제한 병렬은 오히려 대역폭 경합과 서버 rate limit을 유발한다.
기본 4개로 제한하고, 서버가 429/503을 반환하면 지수 백오프로 감속한다.

## 5. 체감 속도 최적화 — Prefetch (기본 활성) ✅ 구현됨 (design.md §4.11)

> **현재 상태 (v1.1)**: prefetch는 **구현되어 기본 활성이다**
> (design.md §4.11). 스케줄러는 5분 주기이며 사용자 작업에 양보한다.
> 트래픽 예산 안전장치는 두지 않는다 — 요금 제약이 없어 폐기됐다(ADR-57).

**대기시간 관점에서는 가장 효과가 큰 전략이지만, 트래픽 관점에서는 순수 이득이 아니다.**
실제 전송 시간을 줄이는 게 아니라 **전송을 사용자가 기다리지 않는 시간대로 옮긴다.**

### 5.0 트래픽 관점의 재평가

prefetch가 소비하는 추가 트래픽은 다음과 같이 나뉜다.

| 구분 | 트래픽 영향 | 설명 |
|------|------------|------|
| 사용자가 결국 fetch할 데이터를 미리 받음 | **중립** | 시점만 앞당길 뿐 총량은 동일 |
| 사용자가 결국 보지 않을 브랜치를 받음 | **손해** | 순수 낭비 |
| 주기 실행으로 인한 협상 왕복 | **소폭 손해** | 변경이 없어도 매번 ref 협상 비용 발생 |

**즉 prefetch의 트래픽 비용은 "사용자가 실제로 무엇을 보는가"에 달려 있고,
이건 측정 없이는 알 수 없다.** 협업이 활발한 저장소에서 대부분의 브랜치 변경을
결국 확인한다면 비용은 거의 0이다. 반대로 브랜치 수백 개짜리 저장소에서
자기 브랜치만 본다면 대부분이 낭비다.

**결정 (v0.9 정정): 기본 활성.** 초안은 기본 비활성 + 온보딩 질문이었다.
그 근거였던 "동의 없이 트래픽을 소비하지 않는다"는 **요금제가 있을 때만 성립한다** — 이 환경에는 없다(ADR-7 정정, design.md §1.4 원칙 4). 대기를 임계 경로 밖으로 옮기는 것이 목적 함수 그 자체이므로 기본으로 켠다. 미룰 이유는 요금이 아니라 배터리·회선 점유다.
UI 설계는 [design.md §5.4](design.md) 참조.

**활성화된 경우의 낭비 최소화 설계**

1. **추적 중인 브랜치만 대상으로 한다.** 전체 `refs/heads/*`가 아니라
   사용자가 최근 조회했거나 체크아웃한 적 있는 브랜치로 범위를 좁힌다.
2. **변경 없으면 즉시 종료.** 협상 단계에서 변경이 없음이 확인되면 팩 전송 없이 끝난다.
3. **적중률을 기록한다.** prefetch로 받은 데이터 중 실제로 사용자가 조회한 비율을
   추적해 설정 화면에 노출한다. 적중률이 낮으면 사용자가 끌 수 있게 한다.

### 5.1 동작 방식

Git의 `maintenance prefetch` 태스크와 같은 원리를 앱이 직접 스케줄링한다.

```bash
git fetch --no-tags --prune --no-write-fetch-head \
    origin +refs/heads/*:refs/prefetch/origin/*
```

**핵심**: 결과를 `refs/remotes/`가 아니라 `refs/prefetch/`에 저장한다.
사용자가 보는 원격 추적 브랜치 상태는 바뀌지 않으므로,
"내가 아무것도 안 했는데 화면이 바뀌었다"는 혼란이 없다.
이후 사용자가 실제 fetch를 누르면 객체는 이미 로컬에 있으므로
**네트워크 전송이 거의 발생하지 않고 즉시 끝난다.**

### 5.2 스케줄링 정책 (활성화한 경우에만)

| 조건 | 동작 |
|------|------|
| 앱 시작 후 30초 | 활성 저장소 1회 prefetch |
| 이후 **5분** 간격 | 활성 저장소 반복 (`_PREFETCH_INTERVAL_MS`, design.md §4.11.4) |
| 사용자 유휴 2분 이상 | 등록된 다른 저장소들도 순차 prefetch |
| 배터리 사용 중 + 20% 미만 | prefetch 중단 |
| 종량제 네트워크 감지 | **prefetch 중단.** 사용자가 명시적으로 해제해야 재개 |
| 창이 백그라운드 | 주기를 2배로 늘림 |
| 월간 트래픽 예산 초과 🔶 | prefetch 중단 후 알림 |

> **정정 (v0.7): 트래픽 예산·상한은 삭제한다.** 근거였던 "트래픽이 제한된
> 환경"이 사실과 달랐다 — 요금제 제약이 없다. 청구서가 없는데 예산을 관리할
> 이유가 없고, 상한이 있으면 오히려 필요한 순간에 작업을 막는다.
>
> prefetch를 미룰 이유로 남는 것은 요금이 아니라 **회선 점유와 배터리**다:
> 사용자가 방금 수동 원격 작업을 시작했으면 양보하고, 배터리 모드면 미룬다.

**사용자 통제권**: prefetch는 설정에서 완전히 끌 수 있어야 한다.
사내망 정책이나 데이터 요금 때문에 원치 않는 사용자가 반드시 존재한다.

### 5.3 낙관적 UI

fetch/push 시작 즉시 UI는 예상 결과 상태로 전환하고,
실패하면 되돌리면서 오류를 표시한다.
사용자는 네트워크 왕복을 기다리지 않고 다음 작업으로 넘어갈 수 있다.

## 6. 로컬 적용 구간 최적화

전송이 끝나도 로컬 처리가 느리면 체감은 그대로 느리다.

### 6.1 commit-graph — 실측으로 강등됨 🔶

```bash
git commit-graph write --reachable --changed-paths
```

커밋의 부모 관계와 생성 시각을 별도 파일에 인덱싱한다.

**Phase 1 실측 정정** ([design.md §4.1.1.3](design.md)): 초안은 "히스토리 로딩이
크게 빨라진다"고 썼으나, **팩된 저장소에서 순회 가속은 측정 오차 수준이다**
(20k 커밋: 4.9 → 4.7µs/커밋). libgit2의 팩 순회가 이미 충분히 빨라
인덱스의 추가 이득이 나타나지 않는다.

남는 잠정 후보는 `--changed-paths`(Bloom 필터)의 **파일 경로 기준 히스토리
조회(`git log -- path`) 가속**뿐이며, 이는 아직 측정하지 않았다.

**적용**: 자동 생성하지 않는다. 경로 히스토리 기능을 만드는 시점에
`git log -- path` 벤치마크로 이득을 확인한 뒤 도입을 결정한다.

### 6.2 파일시스템 모니터

```bash
git config core.fsmonitor true
git config core.untrackedCache true
```

`git status`가 전체 워킹 트리를 스캔하지 않고 파일시스템 변경 알림에 의존하게 한다.
파일 수가 많은 저장소에서 status 소요시간이 수 초 → 수십 ms 수준으로 떨어진다.
클라이언트는 status를 매우 자주 호출하므로 누적 효과가 크다.

### 6.3 인덱스 튜닝

```bash
git config index.version 4          # 경로 접두사 압축으로 인덱스 크기 축소
git config feature.manyFiles true   # 파일 많은 저장소용 프리셋
```

### 6.4 압축 수준: 하한선을 지킨다 (실측으로 정정됨)

전송량과 CPU는 서로를 판다. 일반적인 Git 튜닝 조언은 회선 속도에 따라
최적점이 달라진다고 말한다.

| 회선 | 통상적 권고 |
|------|------------|
| 느린 회선 | 압축 강화 — 전송 시간이 지배적 |
| 빠른 회선 | 압축 완화 — 압축 CPU가 오히려 병목 |
| LAN | 압축 최소 — 대역폭이 사실상 무한 |

**초안의 결론은 "회선과 무관하게 최대로 고정"이었다.** 근거는 §1.1의
목적함수였다 — CPU는 제약이 아니므로 바이트를 1%라도 줄일 수 있다면 CPU를
얼마든지 지불한다는 것. 논리는 맞았지만 **전제가 틀렸다. 살 바이트가 없다.**

#### 6.4.1 실측 — 압축 수준별 push 전송 바이트

파일 120개·커밋 40개의 소스 트리 형태 저장소, 각 수준 3회 중앙값:

| 수준 | 전송 바이트 | 소요 | 기본값 대비 |
|------|------------|------|------------|
| 0 | 142,223 | 0.54s | +113% |
| 1 | 67,000 | 0.55s | +0.6% |
| **6 (기본값)** | **66,591** | 0.54s | — |
| 9 | 66,591 | 0.53s | **+0.00%** |

작은 페이로드에서도 같았다(반복 텍스트 512KB: 6→2,212 / 9→2,243 —
9가 오히려 31바이트 **컸다**). 더 큰 규모와 여러 fixture로 다시 잰 결과는
§6.4.2에 있다 — 6과 9의 차이는 0.00~0.08%이고, 대신 9가 최대 2.35배 느리다.

**해석**: zlib 수준은 객체·델타 각각의 엔트로피 부호화만 바꾸는데, git의
기본값이 이미 6이고 git 객체 크기에서 6→9의 한계 이득은 사실상 0이다.
전송량을 실제로 줄이는 것은 zlib 수준이 아니라 **델타 압축**(`pack.window`,
`pack.depth`)이며 이는 직교한다. 소요시간마저 차이가 없었으므로
"CPU를 주고 바이트를 산다"는 거래 자체가 성립하지 않는다.

#### 6.4.2 그래서 왜 여전히 설정하는가 — 그리고 왜 9가 아니라 6인가

`core.compression=6` / `pack.compression=6`을 명시한다. 목적은 최대화가
아니라 **하한선**이다 — 사용자가 전역 설정에 `core.compression=0`을 넣어둔
경우(빠른 로컬 네트워크를 가정한 흔한 튜닝) 전송량이 2.1배로 뛴다. 명령
단위 `-c`로 덮어써 그 경우를 막는다. `protocol.version=2`를 명시적으로
주는 것과 같은 성격의 방어다.

**한때 이 자리에 9를 적었다.** "하한선이 목적"이라고 적어놓고 값은 최대치를
쓰는 모순이었다.

##### 먼저, 이 자리에 한 번 틀린 숫자를 적었다

처음 재측정했을 때 "9가 만든 팩은 6이 만든 팩과 **바이트 단위로 같다**
(0.00%)"고 적었다. **그것은 측정 방법의 산물이었다.** 전송량을 git의 진행률
문자열(`6.68 MiB`)에서 파싱했는데, 그 표기는 소수 둘째 자리에서 반올림된다 —
같은 `6.68 MiB`로 찍히는 실제 값의 폭이 약 10,486바이트, 즉 **0.15%다.**
6과 9의 진짜 차이는 그보다 작았고, 그래서 "같다"로 보였다.

재감사에서 다른 fixture로는 재현되지 않는다는 지적을 받고 **팩 파일 크기를
직접 재는 방법으로 다시 측정**했다. 반올림이 없는 쪽이다.

| fixture | 수준 | 팩 파일(실측) | 소요(최소) | 6 대비 |
|---------|------|--------------|-----------|--------|
| 반복 많은 소스 트리 0.5MB | 0 | 151,063 | 461ms | +201% |
| | **6** | **50,181** | 463ms | — |
| | 9 | 50,181 | 472ms | 바이트 **±0**, 시간 **+2.0%** |
| 반복 적은 텍스트 2.4MB | 0 | 5,411,312 | 374ms | +275% |
| | **6** | **1,441,786** | 479ms | — |
| | 9 | 1,440,624 | 1,128ms | 바이트 **−0.08%**, 시간 **+135%** |

**결론은 그대로지만 근거가 정확해졌다.** 9는 "차이가 없는" 것이 아니라
**압축이 잘 듣는 내용에서는 차이가 없고, 안 듣는 내용에서는 0.08%를 위해
시간을 2.35배 쓴다.** 목적 함수가 대기 시간(ADR-56)이므로 어느 쪽이든 9를
고를 이유가 없다 — 오히려 최악의 경우가 드러나면서 근거가 강해졌다.
ADR-35(델타 설정 탈락)와 정확히 같은 형태의 거래다.

> **여기서 또 배운 것.** §6.4.1이 "논리가 아니라 크기를 측정해야 했다"고
> 적었는데, 이번엔 **측정 도구의 분해능**을 확인하지 않았다. 0.00%라는 값은
> "차이가 없다"가 아니라 "이 자로는 못 잰다"였다. 자릿수가 의심스러울 만큼
> 딱 떨어지는 결과가 나오면 자를 먼저 의심할 것.

그러면 왜 굳이 6으로 바꿨는가. 성능만이 이유는 아니다. 코드에 최대값이
박혀 있으면 "CPU는 남으니 최대로"라는 이미 두 번 정정된 추론이 계속
되살아난다(ADR-8 정정, ADR-56 정정). 기본값을 적어두면 그 줄을 읽는 사람이
"왜 기본값을 굳이 명시하지"라고 묻게 되고, 그 질문의 답이 바로 하한선 방어다.

> **여기서 배운 것.** "CPU가 남으니 최대로"는 목적함수에서 곧장 따라 나오는
> 결론처럼 보였고, 그래서 측정 없이 통과했다. 실제로는 목적함수가 옳아도
> **한계 이득이 0이면 아무 의미가 없다.** §4.1.1.3의 G2 오판과 같은 유형이다
> — 논리가 아니라 크기를 측정해야 했다. (design.md ADR-8 정정)

| 설정 | 효과 |
|------|------|
| `core.compression=6` / `pack.compression=6` | 기본값과 동일. **0으로 꺼지는 것을 막는 하한선**이 유일한 목적이다 (§6.4.2) |
| ~~`pack.window` 확대~~ | **측정 후 채택하지 않음** (§6.4.3) |
| ~~`pack.depth` 확대~~ | **측정 후 채택하지 않음** (§6.4.3) |

#### 6.4.3 델타 설정 — 측정했고, 채택하지 않았다

zlib 수준이 무의미했을 때 "전송량을 줄이는 실제 지렛대는 델타 쪽일
가능성이 크다"고 적어두고 미뤘던 측정이다. 세 가지 저장소로 쟀다
(각 3회 중앙값, push 전송 바이트).

| 저장소 | `window=250 depth=250` 의 전송량 | 소요시간 |
|--------|-------------------------------|---------|
| 합성 2.9MB | −4.30% | +2% |
| 합성 11.7MB | −2.38% | +20% |
| **실제 저장소**(이 프로젝트 히스토리) | −0.51% | +24% |

**이득은 작고 저장소가 커질수록 줄며, 비용은 반대로 커진다.** zlib 0→1의
130배 같은 것은 없다. 게다가 값이 저장소마다 0.5~4%로 흔들려, 어떤 값을
박아두든 근거가 약하다.

> **합성 fixture가 이득을 과소평가할 수 있다는 점은 인정한다.** 생성기가
> 파일 내용을 무작위로 만들어서, `window`가 노리는 *파일 간 유사성*이
> 현실보다 빈약하다. 그래서 실제 저장소로도 쟀는데 — 오히려 이득이 더
> 작았다. (실제 저장소가 작아 이 역시 결정적이지는 않다.)

**채택하지 않는 이유는 목적함수의 정밀한 적용이다.** §1.4는 "컴퓨팅 파워는
충분하다"고 했지만, 그것이 **사용자의 대기 시간이 공짜**라는 뜻은 아니다.
push 중의 CPU는 사용자가 "올리는 중..."을 보며 기다리는 시간이고, G4가
지키려는 것이 바로 그 반응성이다. 몇 %를 위해 push를 20% 늘리는 것은
목적함수가 사는 트레이드가 아니다.

🔶 **다만 유휴 시 repack에서는 다르다.** 그때는 사용자가 기다리지 않으므로
CPU가 진짜로 공짜다. 델타 설정을 쓸 자리가 있다면 그쪽이다(§9의 9순위).
그건 repack 스케줄링을 만들 때 함께 측정한다.

**주의: 이 설정들이 실제로 효과를 내는 지점은 push다.** fetch/clone에서 팩을
만드는 주체는 **서버**이므로 클라이언트 압축 설정이 다운로드 크기를 바꾸지 못한다.
클라이언트가 다운로드 트래픽을 줄이는 수단은 압축이 아니라 **요청 범위 축소**(§2, §3)다.
이 구분을 흐리면 효과 없는 설정을 켜두고 최적화했다고 착각하게 된다.

| 방향 | 팩 생성 주체 | 클라이언트 압축 설정 효과 | 실제 수단 |
|------|-------------|------------------------|-----------|
| push (업로드) | 클라이언트 | 하한선 방어만 (§6.4.1) | 델타 설정 🔶, 커밋 단위 관리 |
| fetch/clone (다운로드) | 서버 | 없음 | 요청 범위 축소, 협상 최적화 |

**로컬 저장소에도 동일 원칙 적용** 🔶 (미구현 — repack 스케줄러가 없다): 유휴 시간에 `git repack -a -d -f --depth=250 --window=250`을
실행해 로컬 팩을 재구성한다. 디스크와 CPU를 써서 저장소를 조밀하게 유지하면
이후 push 시 전송량도 함께 줄어든다.

## 7. 안티패턴 — 하지 말아야 할 것

인터넷에 널리 퍼진 "git 속도 향상 팁" 중 상당수는 효과가 없거나 해롭다.
이 클라이언트는 다음을 **적용하지 않는다.**

| 흔한 조언 | 왜 하지 않는가 |
|-----------|---------------|
| `http.postBuffer`를 크게 설정 | push 실패 시 만병통치약처럼 퍼져 있지만, 실제로는 전체 요청을 메모리에 버퍼링해 **메모리를 낭비하고 속도를 떨어뜨린다.** 원래 문제는 대개 프록시 설정이나 서버 제한이다. |
| `gc.auto=0`으로 GC 비활성화 | 당장은 빨라 보이지만 느슨한 객체가 쌓여 저장소가 비대해지고 push 전송량까지 늘어난다. 대신 GC를 유휴 시간대로 **스케줄링**한다. |
| 무제한 병렬 fetch | 대역폭 경합과 서버 rate limit을 유발해 총 소요시간이 오히려 늘어난다. |
| 모든 저장소에 shallow clone 강제 | merge-base, blame, push에 제약이 생겨 사용자가 원인 모를 오류를 만난다. 선택지로 제공하되 기본값으로 삼지 않는다. |
| `--depth=1`과 partial clone 동시 적용 | 제약만 중첩되고 이득은 크게 겹친다. 하나를 고른다. |
| 서버 응답 캐싱 프록시 자체 구현 | 정합성 문제와 유지보수 비용이 이득을 압도한다. 서버를 통제할 수 있는 조직에서만 의미가 있다. |
| **클라이언트 압축 설정으로 다운로드 줄이기** | fetch/clone의 팩은 **서버가** 만든다. 클라이언트 압축 설정은 다운로드 크기에 영향이 없다. 켜두고 최적화했다고 착각하기 쉬운 대표적 함정. (§6.4) |
| ~~**체감 속도를 위해 트래픽을 쓰는 기능을 기본 활성화**~~ | **v0.7에서 폐기.** 근거였던 "트래픽이 제한된 환경"이 사실과 달랐다 — 요금제가 없으므로 소비될 사용자 자원이 없다. prefetch는 대기를 임계 경로 밖으로 옮기는 기능이고 그것이 목적 함수 그 자체다 (ADR-7 정정). |
| **초기 전송량만 보고 최적화 판단** | partial clone처럼 초기 전송량은 줄지만 누적 전송량은 늘 수 있는 기법이 있다. 측정 창을 충분히 길게(최소 30일) 잡아야 한다. (§3.1) |

## 8. 측정 체계

**측정 없는 최적화는 추측이다.** 위 기법들의 효과를 데이터로 확인한다.

### 8.1 계측 항목

모든 네트워크 작업에 대해 다음을 기록한다.

모든 네트워크 작업에 대해 다음을 기록한다. ✅ = Phase 3 증분 1에서 구현됨.

| 항목 | 수집 방법 | 상태 |
|------|-----------|------|
| 총 소요시간 | 명령 시작~종료 | ✅ |
| 전송 바이트 | progress 출력 파싱 (`Receiving objects`) | ✅ |
| 객체 수 (received / reused) | git progress 출력 파싱 | ✅ |
| 협상 왕복 횟수 | Trace2 `negotiation_v2.total_rounds` | ✅ |
| 협상된 프로토콜 버전 | Trace2 `transfer.negotiated-version` | ✅ |
| 구간별 소요 (협상/전송/적용) | Trace2 region의 `t_rel` | ✅ |
| 갱신된 ref 수 | progress 출력 파싱 | ✅ |
| **누적 전송량** | 위 값의 롤링 집계. 목적함수의 직접 지표 | ✅ |
| 적용된 최적화 플래그 | 앱이 직접 기록 | 증분 2 |
| RTT | 작업 전 원격 호스트 대상 측정 | 증분 2 |
| **prefetch 적중률** | prefetch로 받은 객체 중 사용자가 실제 조회한 비율 | prefetch 구현 시 |
| **지연 blob 요청 수·바이트** | partial clone 사용 시. 누적 비용 판단 근거 | partial clone 구현 시 |

**실측으로 정정된 부분.** 초안은 전송 바이트와 협상 왕복을
`GIT_TRACE_PACKET`으로 얻겠다고 했으나, 실제로는 두 값 모두 더 싼 경로가
있었다 — 바이트는 progress 출력에, 왕복 횟수는 Trace2가 이미 집계해서
`negotiation_v2` 이벤트로 준다. 패킷 트레이스는 출력량이 크고 파싱이
취약해 채택하지 않았다. 마찬가지로 `GIT_TRACE2_PERF`(사람이 읽는 형식)
대신 `GIT_TRACE2_EVENT`(JSON Lines)를 쓴다.

**측정을 비게 만드는 함정이 하나 있다.** git 기본 설정에서는 작은 fetch가
받은 팩을 풀어버려 `Receiving objects` 줄이 출력되지 않는다 — 즉 가장 흔한
작업인 증분 fetch의 전송량이 조용히 비어버린다. `transfer.unpackLimit=1`로
막는다(design.md ADR-16). 그럼에도 측정이 실패한 경우는 0이 아니라
**측정 안 됨**으로 구분해 기록한다. 섞으면 누적 집계가 과소 보고된다.

**단, "팩을 안 받았다"는 측정 실패가 아니다.** 변경 없는 fetch는 0바이트를
정확히 측정한 것이며, 이걸 미측정으로 세면 일상의 대다수가 미측정으로
분류되어 "측정 공백" 신호 자체가 무의미해진다. 판별자는 `remote: Total ...`
줄의 유무다 (design.md §4.6.1, ADR-19).

**계측을 왜곡하는 경로 셋** — 전부 적대적 리뷰에서 실측으로 확인했다.

| 경로 | 계측에 미치는 영향 |
|------|-------------------|
| 서브모듈 재귀 fetch | 서브모듈 바이트가 누락된 채 **측정 성공으로 기록**된다 (70KB → 329B). 과소 집계가 집계표에서 탐지되지 않는다 |
| 상속된 `GIT_DIR` | 다른 저장소로 fetch가 나가고 바이트는 이쪽에 기록된다 — 과소가 아니라 **오귀속** |
| 같은 저장소의 다른 경로 표기 | 집계 키가 갈라져 누적이 쪼개진다. 키는 정규화된 git 디렉터리를 쓴다 |

앞의 둘은 명령 단위 플래그와 환경변수 필터로 닫았다(design.md §4.6.2).

### 8.2 벤치마크 하네스

`tests/benchmarks/`에 다음을 구성한다.

1. **재현 가능한 대상 저장소**: 커밋 수·파일 수·blob 크기를 파라미터로 하는
   합성 저장소 생성기. 외부 네트워크에 의존하지 않는다.
2. **네트워크 조건 시뮬레이션**: 로컬 서버 앞단에 지연/대역폭 제한을 걸어
   LAN / 광대역 / 원거리 / 저속 4가지 프로파일을 재현한다.
3. **시나리오별 A/B 측정**: 각 최적화를 켠 경우와 끈 경우를 비교한다.

| 시나리오 | 비교 대상 |
|----------|-----------|
| 초기 clone | 전체 vs `blob:none` vs `depth=1` |
| 증분 fetch (커밋 10개) | prefetch 有 vs 無 |
| 반복 fetch (10회) | SSH 멀티플렉싱 有 vs 無 |
| 브랜치 전환 | sparse-checkout 有 vs 無 |
| 경로 히스토리 조회 (`git log -- path`) | commit-graph `--changed-paths` 有 vs 無 — 순회 벤치는 이미 완료(§6.1, 오차 수준) |

### 8.3 성능 회귀 방지

CI에서 벤치마크를 주기 실행하고, 기준 대비 20% 이상 느려지면 실패 처리한다.
성능은 한 번 달성하고 끝나는 게 아니라 **지켜야 하는 속성**이다.

### 8.4 사용자에게 보여주기

> **정정 (v0.7).** 초안은 "이번 달 전송량 / 예산 대비" 대시보드였다.
> **삭제한다** — 요금제가 없으므로 누적 총량은 사용자가 알아야 할 숫자가
> 아니고, 아무도 열지 않는 화면은 대기 시간을 1초도 줄이지 않는다.

보여줘야 할 것은 **지금 얼마나 기다려야 하는가**다. 느린 회선에서 몇 분씩
걸리는 작업에 아무 표시가 없으면 사용자는 앱이 멎었다고 판단한다.

**(가) 진행 중 — 실시간 진행률 (최우선)**

```
받는 중  62%   14.2MiB   1.1MiB/s        ▓▓▓▓▓▓░░░░
```

**남은 시간은 넣지 않는다.** git이 총 바이트를 알려주지 않아 진행률로
역산해야 하는데, 팩이 커밋→트리→blob 순으로 쓰여 객체당 바이트가 뒤로 갈수록
커지므로 그 추정은 구조적으로 낙관 편향이다 (design.md §4.6.5, ADR-59).
git 자신이 속도만 보여주는 것도 같은 이유다.

단계마다 병목이 다르므로(서버 CPU / 회선 / 로컬 디스크) 무엇을 기다리는지
함께 밝힌다 — 그래야 사용자가 "왜 느린가"에 답할 수 있다.
구현 완료 (design.md §4.6.5).

**(나) 끝난 뒤 — 이번 작업의 요약**

지금도 상태바에 전송량을 표시한다. 여기에 소요시간을 나란히 둔다.
누적이 아니라 **이번 한 번**이 판단 근거다.

**(다) 진단용 이력 (선택)**

"이 저장소는 fetch가 평균 40초 걸린다" 정도는 설정을 조정할 때 쓸모가 있다.
다만 이건 예산 관리가 아니라 **진단**이므로, 최근 몇 건이면 충분하다
(현재 (저장소, 종류)별 500건 롤링으로 이미 충분하다 — design.md §13-2의 "30일 누적
비교" 전제는 예산 개념과 함께 폐기됐다).

**설계 의도**: 기다리는 사람에게 필요한 것은 "얼마나 더"이지 "이번 달 총계"가
아니다.

**Phase 3 증분 1의 최소 구현**: 설정 화면은 아직 없고, 상태바에 마지막
원격 작업의 전송량을 상시 표시한다. 임시 메시지로 띄웠더니 fetch 직후의
그래프 재로딩 메시지에 즉시 덮여 사용자가 비용을 볼 수 없었다 — 목적함수를
보여주는 것이 요점이므로 고정 위젯으로 옮겼다(회귀 테스트 있음).

### 8.5 구현 현황 (Phase 3 증분 1~4)

| 구성요소 | 위치 |
|----------|------|
| progress·Trace2 파서 (순수 파이썬) | `domain/instrumentation.py` |
| git CLI 실행·환경 격리, fetch/push/clone | `infrastructure/remote_engine.py` |
| 병합 분석·빨리 감기·ahead/behind (pygit2) | `infrastructure/local_engine.py` |
| 자격증명 공급 shim | `infrastructure/askpass.py` |
| 이력 저장·집계 (SQLite, 저장소별 500건 롤링) | `infrastructure/stats_store.py` |
| fetch/push/pull/clone 워커 | `application/remote_workers.py` |
| 원격 통합 테스트 하네스 (`file://`) | `tests/integration/remote_harness.py` |
| 인증 테스트 하네스 (Basic 인증 HTTP) | `tests/integration/auth_harness.py` |

하네스는 원격 URL로 **`file://`를 쓴다.** 일반 로컬 경로를 주면 git이
팩 프로토콜을 건너뛰는 지름길을 타서 `Receiving objects`가 나오지 않는다 —
즉 측정 경로가 테스트에서 실행되지 않는다.

§8.2의 네트워크 조건 시뮬레이션과 §8.3의 CI 회귀 감시는 **여전히 없다.**
증분 2~4가 기능 축으로 진행되면서 미뤄졌다.

**계측이 값을 돌려준 지점 둘**:
- 증분 2에서 압축 수준을 측정해 ADR-8이 근거 없는 설정임을 확인했다(§6.4.1)
- 증분 4에서 델타 설정(`pack.window`/`pack.depth`)을 측정해 채택하지 않기로
  했다(§6.4.3, ADR-35)

둘 다 **하지 않기로 한 결정**이라는 점이 요점이다. 계측이 없었으면 둘 다
"이론상 이득이니 켜두자"로 남았을 것이다.

## 9. 구현 우선순위

> 🔶(미구현) 항목의 현재 우선순위는 **[backlog.md §4](backlog.md)**에 있다.
> 이 표는 "무엇이 얼마나 효과가 있는가"의 기록이고, 그쪽은 "다음에 무엇을
> 할 것인가"의 목록이다.


**정렬 기준은 "사용자 대기 시간 단축 ÷ 구현 비용"이다** (§1.1의 목적함수).

> **정정 (v0.7).** 이 표는 "전송 바이트 절감 ÷ 비용"으로 정렬돼 있었다.
> 바이트가 여전히 주된 지렛대이므로 순위 대부분은 유지되지만, 두 가지가
> 달라진다: **임계 경로 밖 작업(prefetch)의 순위가 오르고**, CPU를 크게 쓰는
> 항목(델타 설정)은 절감이 있어도 탈락한다.

| 순위 | 항목 | 트래픽 절감 | 비용 | 비고 |
|------|------|------------|------|------|
| 1 | 계측 체계 (Trace2 파싱) | — | 낮음 | ✅ **완료** (증분 1). 측정 없이는 나머지의 효과를 알 수 없다 |
| 2 | protocol v2 명시 | 중~대 | 매우 낮음 | ✅ **완료** — 협상된 버전을 계측으로 확인 중 |
| 3 | ref 요청 범위 축소 | 중~대 | 낮음 | 🚧 fetch는 `--no-tags` 기본. 명시적 refspec은 API만 있고 UI 미연결 |
| 4 | ~~압축 최대 고정 (push)~~ | ~~중~~ **없음** | 매우 낮음 | ⚠️ **실측으로 강등** — 기본값 대비 0.00%. 하한선 방어로만 유지 (§6.4.1) |
| 5 | SSH 멀티플렉싱 | 중 (왕복 제거) | 낮음 | 플랫폼 감지 필요 |
| 6 | 협상 알고리즘 튜닝 | 중 (원거리) | 낮음 | RTT 기반 자동 적용 |
| 7 | 중복 요청 억제 | 중 | 중 | 앱 레벨에서 동일 fetch 요청 병합 |
| 8 | core.fsmonitor | — (로컬 가속) | 낮음 | status 호출이 잦아 누적 효과 |
| 9 | 로컬 repack 스케줄링 🔶 | 소~중 | 낮음 | 유휴 CPU 활용. **델타 설정을 쓸 자리가 여기다** — push와 달리 사용자가 기다리지 않는다 (§6.4.3) |
| 10 | sparse-checkout | — (디스크·checkout) | 중~대 | 트래픽 절감 아님. 모노레포에서만 |
| 11 | commit-graph (`--changed-paths` 한정) 🔶 | — (경로 조회 가속, 미측정) | 낮음 | **순회 가속은 실측 오차로 판명(§6.1)** — 초안 8위에서 강등. 경로 히스토리 기능 도입 시 벤치 후 결정 |
| 12 | partial clone 🔶 | 초기 대 / **작업 중 정지 위험** | 중 | 선택지로만. 기본 비활성 (ADR-6 확정) |
| 13 | 백그라운드 prefetch | **대기를 임계 경로 밖으로 — 실측 다음 fetch 전송 0** | 중 | ✅ **구현 완료** (기본 활성, ADR-7 확정). 요금이 없으므로 총량 증가는 비용이 아니다 |
| 14 | bundle URI | 상황에 따라 대 | 낮음 | 지원 서버 제한적. 기회적 적용 |

**Phase 3 착수 시 1~6번을 함께 적용한다.** 비용이 낮고 전송량 절감이 확실하다.
증분 1에서 1·2번을 완료하고 3·4번을 fetch 경로에 적용했다.
**5~7번(SSH 멀티플렉싱, 협상 튜닝, 중복 요청 억제)은 증분 4까지도 미구현이다** —
증분 2~4가 push/pull·인증·복제라는 기능 축으로 진행되면서 뒤로 밀렸다.
비교할 작업이 여러 개 갖춰진 지금이 측정하기 좋은 시점이다.

**12~13번은 계측 데이터가 쌓인 뒤에 판단한다.** 트래픽 관점에서 이득인지가
아직 검증되지 않았다.

### 9.1 순위 변경 이력

| 항목 | 초안 순위 | 현재 | 변경 사유 |
|------|----------|------|-----------|

## 10. 요약

이 설계의 통신 최적화 전략은 네 문장으로 요약된다.

1. **안 받아도 되는 것은 안 받는다** — ref 범위 축소, protocol v2, 중복 요청 억제.
2. **받은 것은 두 번 받지 않는다** — 전체 clone 기본값, 공격적인 로컬 캐싱.
   오프라인 작업이 잦다는 조건이 여기에 힘을 싣는다.
3. **연결 비용은 한 번만 낸다** — SSH 멀티플렉싱, HTTP 커넥션 재사용.
4. **CPU로 바이트를 산다 — 단, 살 바이트가 있을 때만.** 이 문장은 초안에서
   가장 크게 수정된 부분이다. 압축 최대 고정은 기본값 대비 **0.00%**였고
   (§6.4.1), 델타 설정은 0.5~4%에 소요시간 +20%였다(§6.4.3). 목적함수가
   옳아도 **한계 이득이 0이면 거래가 성립하지 않는다.** 게다가 "컴퓨팅이
   남는다"가 "사용자의 대기 시간이 공짜"라는 뜻도 아니다 — push 중의 CPU는
   사용자가 기다리는 시간이다. CPU를 쓸 자리는 사용자가 기다리지 않는
   유휴 시간이다.

**그리고 이 모든 것의 전제는 측정이다.**

이 문서의 두 가지 주요 결정 — 전체 clone 기본값(§3.1)과 prefetch 옵트인(§5) — 은
"트래픽 총량을 최소화한다"는 목적함수에서 논리적으로 도출한 것이지 실측으로
검증된 것이 아니다. 두 결정 모두 🔶로 표시했으며, Phase 3의 계측 데이터가
논리와 다른 결과를 보이면 **주저 없이 뒤집는다.**

측정해야 할 핵심 질문 두 가지:

- partial clone의 **30일 누적** 전송량은 전체 clone보다 정말 많은가?
- prefetch가 받은 데이터 중 사용자가 실제로 조회하는 비율은 얼마인가?

이 두 숫자가 나오기 전까지 위 결정은 **잠정안**이다.

> **Phase 3이 끝난 시점의 정직한 상태**: 두 질문 모두 **아직 답이 없다.**
> 첫 번째는 장기 사용 데이터가 필요한데 계측 체계만 갖춰졌을 뿐 데이터가
> 없고, 두 번째는 prefetch 자체가 미구현이라 잴 대상이 없다. Phase 3에서
> 실측으로 뒤집힌 것은 이 둘이 아니라 압축(§6.4.1)과 델타 설정(§6.4.3)이었다.
> 계측이 있으면 **묻지 않은 질문에도 답이 나온다**는 것이 이번 교훈이다.
